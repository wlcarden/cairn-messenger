# §§6/7 — Solo-Developer Feasibility Lens

## Summary

§6.1 packages the v1 scope as "what one developer can ship soundly in 9–12 months" (line 567), but the brief simultaneously establishes (a) a volunteer "as available" cadence per D0008, dropping calendar predictability, and (b) §8.1 lists three distinct specialist roles — cryptographic consulting, UX engineer, documentation/community — whose work §6.1 then re-absorbs into the solo developer's plate. The 9–12 month estimate appears bound to a calendar a volunteer baseline cannot back. D0004 explicitly cites the Section 5 review's engineering estimate of 18–30 months at full scope; the brief credits four scope cuts as sufficient to return to 9–12 months, but the cuts retired by D0004 (Briar, repro builds, CRDT, three UX surfaces) do not retire the multi-specialization cross-section §6.1 still requires.

The acute solo-feasibility risks concentrate in three places: (1) §6.1 bundles Rust crypto core, Android Keystore integration, Kotlin UI, and UniFFI binding maintenance into one developer's working memory — D0003's Rust/Kotlin boundary itself requires sustained bilingual context across the whole project lifetime; (2) §6.3 commits the developer personally to facilitator throughput for 10–15 pilot users at the threat tier, in parallel with implementation; (3) §7 v1.5 and v1.6 contain the deferred UX work §8.1 explicitly identified as "30-50% of v1 total" and as the UX engineer's natural scope, but the roadmap does not name the funding precondition for v1.5/v1.6 the way §10 names it for the pre-pilot audit.

## Critical findings

### F1: §6.1 v1 deliverable bundle spans four specializations §8.1 names as separate roles

- **Evidence:** §6.1 (lines 567–583) lists, as the solo developer's v1 output: (a) Rust cryptographic primitives wrapper plus Shamir reconstruction with memory hygiene plus COSE_Sign1 capability tokens plus trust-graph operation envelope and Sigsum client per D0003 and D0006; (b) Kotlin UI with Signal-familiar surface, trust-badge rendering, recovery-flow walkthroughs, and calibrated security-communication per §5.6; (c) external reviewer-pool recruitment, onboarding, and per-release coordination per §5.5 and §8.2; (d) facilitator handbook, peer-recovery handbook, post-coercion written guidance, security-model overview per §6.1 Documentation. §8.1 (lines 724–726) names three of these four as funding-gated specialist roles: "Part-time cryptographic consulting" for the design-and-implementation cryptographic engineering, "UX-focused engineer" for §5.6 ("30–50% of v1 total per the engineering review"), and "Documentation and community-management role" for §5.7-scope documentation work. §6.1 silently re-absorbs all three into one developer.
- **Impact:** This is the structural feasibility question §6.1 does not answer. The brief's own §8.1 estimate is that the §5.6 UX surface alone is 30–50% of v1 effort, and §10.3 budgets a "3-6 month engagement" UX specialist for it. If §8.1's own estimate is directionally correct, the solo developer either takes 6–10 months on UX alone (consistent with the 18–30 month full-scope estimate in D0004 context) or the UX ships at materially lower quality than §5.6 specifies (subtle trust signals, warning-fatigue-avoiding calibration, recovery-flow walkthroughs designed for users under coercion stress). The latter is a security property regression, not a feature compromise: §5.6 line 514 frames "friction in security software translates directly to operational mistakes" — UX quality at this threat tier is in the security boundary.
- **Recommendation:** §6.1 should explicitly name which §8.1 specialist work the solo developer absorbs, at what quality bar, and where the brief acknowledges the gap. The current "what one developer can ship soundly" framing implies the bundle is solo-feasible; the §8.1 listing implies it is not. One of those needs to give. Candidate framings: (a) name §5.6 UX as "MVP-quality, deferred to UX engineer for v1.5 polish"; (b) move the documentation deliverable to "intent subject to partner co-production" rather than committed solo output; (c) name the reviewer-pool coordination work as separate from implementation effort with its own time budget.

### F2: §6.3 pilot facilitation is a continuous parallel commitment, not a one-time cost

- **Evidence:** §6.3 (lines 625–632) commits the developer personally to in-person provisioning ceremony facilitation for 10–15 pilot users, covering identity generation, recovery peer designation, pre-shared challenge establishment "with each peer", initial trust-graph seeding, and walkthrough of post-coercion guidance. Each pilot user has 3–5 recovery peers per §5.3's default; with 10–15 users that is 30–75 peer-side challenge establishments, each requiring out-of-band coordination per §5.3 line 404. §6.3 line 634 commits a six-month pilot duration "before any broader release decision" — meaning the developer holds facilitator role through the full pilot, in parallel with whatever v1.5 implementation work has begun. §5.6 line 530 makes facilitator presence "the unit of operational discipline" — i.e., the facilitator's continued availability is a security property of the deployment, not a launch-day operation.
- **Impact:** The brief does not estimate facilitator-hours per pilot user or aggregate facilitator hours across the pilot. Reasonable estimate from §5.3 + §6.3 + §5.6: 8–16 hours per user for the initial ceremony (selection guidance walkthrough, peer-by-peer challenge establishment with out-of-band channels, trust-graph seeding) plus 2–4 hours per user per quarter for follow-up (peer rotation events, recovery scenarios per §6.3 line 634, support-channel issues per §5.7) — 120–360 hours of facilitator work in pilot calendar time. At the volunteer "as available" cadence per D0008, this is a substantial portion of available development time. The brief's §10.1 Phase A risk paragraph (line 1053) acknowledges "developer burnout extends Phase A indefinitely or terminates it prematurely" but does not acknowledge that the facilitator role is itself a sustained burnout vector during the pilot phase the brief commits to.
- **Recommendation:** §6.3 should either (a) name a co-facilitator role (partner organization staff, recruited volunteer facilitator) and acknowledge facilitator throughput as the binding constraint on pilot scale rather than "10–15 in one or two local groups", or (b) name the facilitator-hour estimate explicitly and acknowledge it as a draw on the developer's volunteer time that competes with v1.5 implementation. The current framing — "at pilot scale the developer can sustain the facilitator role personally" — is asserted without a time-cost grounding.

### F3: D0003 UniFFI binding maintenance is a continuous bilingual context cost §6.1 does not budget

- **Evidence:** D0003 (lines 16–35) commits the project to a Rust core plus Kotlin UI architecture with UniFFI bindings. D0003 line 78 acknowledges "Build tooling adds a Rust toolchain dimension. The reproducible-build pipeline (when added in v1.5 per D0004) must reproduce both the Rust artifact and the Kotlin artifact." D0003 line 79 splits testing across three test infrastructures: Rust unit/property-based tests, Kotlin tests for UI, and integration tests at the binding boundary. §5.7 line 540 reiterates this split. The UniFFI binding layer itself is named as a per-feature ongoing cost in D0003 line 14: "increases it slightly for others (UniFFI binding layer)". §6.1 does not name UniFFI binding maintenance as a deliverable category.
- **Impact:** UniFFI binding regeneration is required on any Rust public-API change, which at v1 includes capability-token construction, trust-graph operations, recovery-flow primitives, and Shamir reconstruction (all named in §10.1 Phase A line 1029). The developer must hold simultaneous Rust ownership-model context and Kotlin/Android-lifecycle context across every iteration. This is not a one-time setup cost; it is a sustained cognitive switching cost that compounds when the developer returns to either side after a gap. At "as available" cadence per D0008, gaps between sessions are expected. The maintenance overhead the brief acknowledges in D0003 line 14 (small per-change cost) compounds across the calendar duration the volunteer baseline implies, and is borne entirely by one developer who must remain current on both stacks.
- **Recommendation:** §6.1 should acknowledge UniFFI binding maintenance and the bilingual-context-switching cost as part of the developer's v1 working set, not as a one-time D0003 setup. The §10.1 Phase A deliverables list (line 1029) names "Rust core implementation per D0003" and "Kotlin Android UI via UniFFI" as separate line items but does not name binding-layer maintenance, integration testing at the boundary, or the Rust↔Kotlin reproducible-build coordination D0003 itself flagged. These should appear as explicit deliverables so reviewers and successors (per §8.1 line 720) understand the actual surface area.

## Significant findings

### F4: §7.1 v1.5 and v1.6 timing implies continued solo cadence the brief does not justify

- **Evidence:** §7.1 (line 670) splits v1.5 into "architecture-completeness" (v1.5, ~6–9 months post-v1) and "deferred-UX" (v1.6, ~12–15 months post-v1) "to reflect honest engineering scope per the Sections 8/9 review F9". D0008 line 36 acknowledges that volunteer-baseline cadence implies "v1.5 ships at 10–18 months post-v1 launch rather than the 6 months the brief currently states". §7.1 line 670 then says "The split assumes the volunteer-baseline release cadence per D0008; funded-state cadence and team growth would compress the timeline correspondingly." But §10 names no Phase B/C funding precondition for v1.5/v1.6 the way it names the Phase B audit precondition for pilot launch — v1.5/v1.6 work appears in the roadmap as if it proceeds whether or not §8.1 specialist funding closes.
- **Impact:** v1.5 includes Briar integration (D0004 line 20 acknowledges as "multi-month work that v1 cannot absorb"), reproducible Android builds (§5.5 line 482: "4–8 weeks of toolchain work plus ongoing maintenance"), and per-D0004 §6.2 the v1.6 release adds local trust-graph caching, in-app post-coercion flow, multi-profile UX, duress-wipe, voice/video, and localization. This is materially more specialization breadth than v1 (mesh and iOS land in v2; v1.5/v1.6 is where Briar's Tor pluggable-transport plumbing intersects with the developer's existing surface). The brief does not name whether v1.5 proceeds at volunteer baseline if §8.1 specialist funding has not closed by then. The honest answer is that v1.5 at volunteer baseline plus pilot facilitation per §6.3 plus reviewer-pool coordination per §8.2 may not be deliverable on the timeline §7.1 suggests, even with the v1.5/v1.6 split D0008 added.
- **Recommendation:** §7.1 should add a Phase B/C funding-state contingency for v1.5/v1.6 analogous to §10.3 Phase C ("If Phase C funding closes at the lower bound only..."): name explicitly whether v1.5/v1.6 ships at volunteer baseline, ships at funded baseline only, or ships in degraded form at volunteer baseline. The current framing leaves the question implicit.

### F5: §6.1 Tor pluggable-transport posture requires sustained Tor-ecosystem context the developer may not hold

- **Evidence:** §5.4 lines 452–454 commit the project to "tracking Tor Project transport guidance and prioritizing transport-update releases when active blocking is observed in pilot or partner jurisdictions". The transport stack named — obfs4, meek, webtunnel, snowflake — is a continuously evolving DPI-evasion problem ("transport choices appropriate at v1 release may be blocked by v2"). §6.1 incorporates this commitment by virtue of "SimpleX runs over Tor in v1" but does not name pluggable-transport tracking as a v1 maintenance responsibility. The arti integration named in D0003 line 24 is "via arti where viable" — arti's pluggable-transport story is itself evolving and may not match what `tor` provides at v1 ship time.
- **Impact:** Pluggable-transport tracking and rapid-response transport-update releases require: (a) Tor Project ecosystem fluency, (b) the ability to integrate and test new transports against the messaging stack, and (c) the ability to ship a release on emergency-path cadence per §8.2 line 738 when active blocking is observed in pilot jurisdictions. The emergency-release path requires 2-of-5 reviewer attestation within whatever timeline the threat in question imposes. At solo + volunteer reviewer baseline this is a meaningful operational risk: pilot users in DPI jurisdictions are offline until the transport-update propagates through reviewer attestation and distribution. The brief acknowledges the user-side cost (§5.4 line 454: "users in DPI jurisdictions are offline when an active transport is blocked") but does not acknowledge the developer-side specialization requirement.
- **Recommendation:** Either name a Tor-ecosystem-fluency dependency explicitly (e.g., "transport tracking depends on the developer's existing or acquired familiarity with the Tor pluggable-transport ecosystem; if that familiarity is not held, the project commits to partner-organization advisory for transport selection"), or defer the transport-tracking commitment to v1.5 when funded consulting may be available.

### F6: §6.1 Android Keystore / StrongBox integration is muscle memory the brief does not confirm

- **Evidence:** §5.1 line 307 commits the operational identity to "the StrongBox security level backed by the Titan M2 secure element where Ed25519 is supported on the target Pixel generation, TEE-backed otherwise". This requires Android Keystore API fluency, StrongBox capability detection, fallback handling across Pixel generations (Titan M2 is Pixel 6+; earlier targeted Pixels have different secure-element generations), and the hardware-element policy gate work for capability-token signing. §5.7 line 552 also names "Android Keystore-managed key material" as one of the v1 persistent stores. D0003 confines this to the Kotlin UI side (line 33: "Android Keystore API calls"), meaning the developer must handle the Android-side secure-element integration in Kotlin while the Rust core does the cryptographic work — a boundary that requires careful handling of the operational identity's lifecycle across the Rust↔Kotlin interface.
- **Impact:** Android Keystore + StrongBox is not the kind of API surface most non-Android-security developers carry context for. Edge cases around Ed25519 vs. ECDSA availability per Pixel generation, BiometricPrompt integration for the LSKF-gated signing per §5.1 line 307, attestation-key handling, and StrongBox-unavailable fallback are all places where the v1 design's security properties (compelled-unlock bounded exposure per §5.1) depend on correct integration. The brief asserts the design but does not confirm the developer has the muscle memory to implement it without acquiring it during the v1 timeline. If the developer is acquiring Keystore expertise during v1, that is additional calendar cost the 9–12 month estimate does not budget.
- **Recommendation:** §6.1 should either name Android Keystore / StrongBox integration as a known-strength of the developer (confirming the muscle memory exists) or name it as a learning-curve dependency that contributes to the v1 timeline. §10.1 Phase A line 1042 names "Two to four GrapheneOS-capable Pixel devices retained by the project for developer testing" — appropriate, but does not establish that Keystore-specific integration testing is part of the budgeted work.

## Minor findings

### F7: §6.1 "Single-profile only in v1" may be cheaper to defer than the brief implies

- **Evidence:** §6.1 line 577 commits to "Single-profile only in v1; multi-profile compartmentation UI is deferred to v1.5". §5.6 line 520 reiterates: "Multi-profile support is architecturally present (per-jurisdiction, per-role, per-organization), but the v1 UI ships single-profile-only per D0004". D0004 cuts this from v1, but §6.4 line 642 still commits to "Protocol versioning fields in all signed messages from day one" and other forward-compatibility plumbing that must work for the v1.5 multi-profile case. The developer must therefore design the v1 storage and identity schemas to accommodate v1.5 multi-profile without shipping the UI for it.
- **Impact:** Schema-level forward-compatibility for an unshipped feature is a known source of solo-developer drift: the design must hold context for the future requirement across the v1 implementation window, and the absence of an implemented consumer makes the schema choices unvalidated until v1.5. This is the kind of small cost that compounds at solo scale.
- **Recommendation:** Acknowledge in §6.1 that v1.5 multi-profile imposes v1 schema-design constraints the developer must hold context for during v1 implementation; treat as a small forward-compatibility tax rather than an unbudgeted cost.

### F8: §6.3 partner-organization debrief cadence is solo-developer coordination work

- **Evidence:** §6.3 line 636 commits to "Partner organization debriefs at the 3-month and 6-month marks (per Section 8.6) provide external perspective on the pilot's progress." This requires the developer to schedule, prepare for, conduct, and document partner debriefs in addition to facilitator and implementation work.
- **Impact:** Small in isolation, additive to the §6.3 facilitation throughput concern in F2. Partner debriefs are unbudgeted coordination time competing with v1.5 implementation per §7.1.
- **Recommendation:** Either fold debrief coordination into the facilitator-time budget recommended in F2, or note explicitly that partner debrief preparation is itself a solo developer responsibility during the pilot window.

## Patterns

**P1: §6 specifies architectural ambition; §8.1 lists the specialist roles that ambition normally requires; §10 phases the funding that would close those roles; §6 does not name which phase delivers what.** F1 and F4 are the same pattern at different time horizons. §6.1 implicitly assumes all §8.1 work folds into the solo developer at v1; §7.1 implicitly assumes the same at v1.5/v1.6. §10 names the funding conditionals for the pre-pilot audit and the foundation-incorporation path but does not name a funding conditional for §8.1's specialist-role transitions. The brief is honest about audit-funding gating (no skip the audit, per §10.2 line 1077 and §10.3 line 1098); it is less honest about specialist-role-funding gating for the v1 and v1.5 implementation surface.

**P2: §6.1's "what one developer can ship soundly" framing inherits a calendar D0008 already invalidated.** The 9–12 month estimate in §6.1 line 567 predates D0008's explicit acceptance that volunteer baseline implies slippage; D0008 line 36 already acknowledges v1.5 timing slips to 10–18 months. The §6.1 estimate has not been comparably updated. Either §6.1 should note that the 9–12 month figure assumes a non-volunteer cadence (and is therefore conditional, not unilateral), or the figure should be revised to the "as available, completes when it completes" framing §10.1 line 1024 uses for Phase A.

**P3: The facilitator role is undercosted across §6.3, §5.6, and §8.6.** §5.6 line 530 makes facilitator presence a security property. §6.3 commits the developer personally to it. §8.6 (not read in full for this review but referenced) presumably names partner facilitators as a v1.5+ pattern. The brief does not establish facilitator throughput as a binding constraint on pilot scale at v1; it asserts 10–15 users is sustainable without naming the time cost. F2 surfaces this; the pattern is that facilitator work is treated as a one-time launch cost rather than a continuous parallel commitment through the six-month pilot window.
