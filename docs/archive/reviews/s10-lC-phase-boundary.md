# §10 — Phase Boundary Discipline Lens

## Summary

§10's phase model is structurally honest at the macro level (Phase A unilateral / Phase B first-dollar / Phase C broader-release / Phase D steady-state) but contains several boundary leaks that undermine its central promise to funders: "what does my dollar unlock?" The most consequential leaks are dependency inversions — items placed in a later phase whose successful completion requires work or money committed in an earlier phase, with the dependency unacknowledged. Foundation incorporation (Phase C, line 1087) depends on legal-consultation cycles that begin in Phase B (line 1061); the pre-incorporation legal consultation itself is filed under Phase B but explicitly described as developer-absorbable (line 1061) — straddling the A/B boundary. The "rolling cryptographic consulting" Phase B aspiration (line 1073) is described as covering "design-and-implementation phase," which is Phase A work by §10.1's definition (line 1029).

Beyond the dependency leaks, §10 contains at least two items whose phase placement does not match the operating-reality framing elsewhere in the brief. Reviewer honoraria are Phase C in §10.3 (line 1086) but §8.2 (line 740) makes "partnership or grant funding closes" — i.e., Phase B — the trigger. The Phase D "recurring audit cycle" (line 1106) is amortized into steady-state operating cost, but the audit itself is a discrete pre-broader-release-style event 18-24 months after the first; treating it as Phase D smear hides its phase-gate character. Together these create ambiguity about exactly what the first $17K, $60K, $105K, or $335K of closed funding actually delivers.

## Critical findings

### F1: Pre-incorporation legal consultation straddles Phase A/B boundary with no resolution rule

- **PROBLEM:** §10.2 places "pre-incorporation legal consultation" as a Phase B line item (line 1061: "$1,500-9,000") but the same sentence states "the consultation is small enough that the developer may absorb it personally if grant timing requires; the brief identifies it as Phase B by category." §10.5 reinforces the ambiguity (line 1123): "Self/personal. Phase A and as long as feasible into Phase B if the developer can absorb the initial legal consultation." §10.6 sequencing rule #2 (line 1146) further compounds it: "The legal consultation may be self-absorbed if the developer can sustain the cost, or grant-funded if a smaller flexible grant closes first."
- **EVIDENCE:** docs/design-brief.md lines 1048 (Phase A "cannot deliver" includes "Pre-incorporation legal consultation"), 1061 (Phase B item, developer may absorb), 1123 (Phase B if absorbed), 1146 (either route).
- **IMPACT:** A funder asking "does my Phase B grant pay for legal consultation?" gets no clean answer. If the developer absorbs it, it is implicitly Phase A but §10.1's $1,500-3,500 absorbed total (line 1043) excludes it. If grant-funded, it is Phase B but the floor calculation in line 1068 assumes it is included ($1,500 lower bound). The brief cannot have it both ways: either Phase A's absorbed total is wrong (~$3K-13K, including consultation) or Phase B's floor is contingent on the developer choosing not to absorb. The phase model's promise of "Phase A = unilateral commitment, Phase B = first conditional" is broken at the seam.
- **RECOMMENDATION:** Pick one of: (a) explicitly make legal consultation Phase A with $1,500-9,000 added to the absorbed total and an honest note that the developer's self-funding capacity governs whether it happens, (b) explicitly make it Phase B and remove the "developer may absorb" escape clause, or (c) introduce a "Phase A+" or "Phase A/B straddle" category for the one item that genuinely lives between gates. A funder needs to know whether a $5K flexible grant unlocks anything material; the current framing makes it impossible to answer.

### F2: Foundation incorporation (Phase C) has unstated dependencies on Phase B legal consultation

- **PROBLEM:** §10.3 places "foundation incorporation" as a Phase C item at $5K-25K (line 1087). The cost is described as "Includes legal counsel for the incorporation filing, registration fees, initial board-governance documents, and IP-assignment transfer mechanics." But §8.4 (line 768, line 787-794) describes the pre-incorporation legal consultation as covering "jurisdictional shortlist evaluation" and "IP-ownership and assignment mechanics for the natural-person-to-foundation transition" — which is the substantive work for incorporation. The Phase B legal consultation (line 1061) is what selects the jurisdiction; without it, the Phase C $5K-25K cannot fund "incorporation" because there is no incorporation target to file for.
- **EVIDENCE:** docs/design-brief.md line 1061 (Phase B consultation: "fiscal-sponsor selection support, jurisdictional shortlist evaluation, and IP-ownership/assignment considerations for the natural-person-to-foundation transition"); line 1087 (Phase C incorporation includes "IP-assignment transfer mechanics"); §8.4 line 787-794 (the diligence considerations all live in legal consultation, not in the incorporation filing itself).
- **IMPACT:** If Phase B closes but Phase C does not, the project has paid for consultation that produces no artifact. If Phase C closes but Phase B did not (or the consultation was self-absorbed under §10.6 #2), the consultation has not happened and the Phase C dollars cannot buy incorporation. The two items are sequentially dependent in a way the phase split obscures. A funder considering a Phase C grant should know that incorporation requires Phase B work to have already landed.
- **RECOMMENDATION:** Either (a) split foundation incorporation into "Phase B+: jurisdictional evaluation (subsumed in legal consultation, no incremental cost)" and "Phase C: filing and transfer mechanics ($5K-25K)" with the sequencing made explicit, or (b) reframe Phase C's foundation incorporation line as "requires Phase B legal consultation to have completed" so the dependency is on-page. The Phase C floor (line 1093, $105K) silently assumes Phase B consultation has happened; this assumption needs to be stated.

### F3: Reviewer honoraria phase placement contradicts §8.2's funding-trigger framing

- **PROBLEM:** §10.3 places "reviewer honoraria" as a Phase C item (line 1086: $40K-100K/year, "for the post-pilot operational period"). But §8.2 (line 740) frames the volunteer-to-honoraria transition as triggered by funding generally: "Honoraria become the operational model once partnership or grant funding closes (Q3)." Q3 funding closing is what Phase B represents (per the brief's §10 framing) — Phase B is the first grant intake. §8.2 (line 736) also describes "partner-funded honoraria operations" as targeting "quarterly cadence," which is described as a pilot-period operational change, not a post-pilot one. The pilot period is Phase B by §10's gating.
- **EVIDENCE:** docs/design-brief.md line 740 ("Honoraria become the operational model once partnership or grant funding closes"); line 736 (cadence shifts under honoraria during the pilot/v1 period); line 1075 (Phase B unlocks pilot deployment); line 1086 (Phase C honoraria are "for the post-pilot operational period"); line 1049 (Phase A says honoraria "require grant funding (Phase B/C)" — itself ambiguous between B and C).
- **IMPACT:** §8.2 promises reviewers honoraria when "the project can sustain it" — naming Phase B grant outcomes as the trigger. §10.3 places the budget line in Phase C. A reviewer evaluating recruitment under §8.2's surfacing commitment cannot determine whether honoraria arrive on Phase B funding or only on Phase C funding. The brief's "volunteer-to-honoraria-to-volunteer" possible trajectory (line 740) becomes incoherent if Phase B closes (some reviewers expect honoraria per §8.2) but Phase C does not (no honoraria line item budgeted). The funding-state-surfacing commitment requires phase clarity that the current placement undermines.
- **RECOMMENDATION:** Split reviewer honoraria into "Phase B: pilot-period honoraria for the duration of pilot operations (sized at pilot scale: perhaps $5K-20K total for one release cycle)" and "Phase C: full operational-model honoraria ($40K-100K/year, post-pilot quarterly cadence)." Or, if the intent is that pilot reviewers remain on volunteer-attestation throughout pilot regardless of Phase B funding (i.e., Phase B's audit-funding does not unlock honoraria), say so explicitly in §10.2 and reconcile §8.2 line 740.

### F4: "Rolling cryptographic consulting" Phase B aspiration covers Phase A work

- **PROBLEM:** §10.2 line 1073 names rolling cryptographic consulting "at $5-10K/month for 3-6 months of design-and-implementation phase" as a Phase B aspiration. But "design-and-implementation phase" is Phase A by §10.1's definition (line 1029, "Rust core implementation per [D0003]: cryptographic primitives wrapper..."). §8.1 line 724 confirms: rolling consulting is intended to operate "rolling review during implementation." If Phase B funding closes after substantial Phase A implementation is complete (the brief does not commit to a Phase A calendar, line 1024), the rolling-consulting window has already largely closed and the funding event misses its leverage point.
- **EVIDENCE:** docs/design-brief.md line 1024 (Phase A "as available" cadence, no calendar commitment); line 1029 (Phase A includes the cryptographic implementation work); line 724 (rolling consulting is during "design-and-implementation phase"); line 804 (consultant "reviews design decisions as they are made"); line 1073 (Phase B aspiration covers same "design-and-implementation phase").
- **IMPACT:** The Phase B framing implies rolling consulting is unlockable by Phase B funding outcomes, but the design-and-implementation window may have closed before Phase B funding lands — Phase A's open-ended timeline means by the time a grant cycle (3-9 months per §10.6 #3) closes, the cryptographic primitives are likely already implemented and the consultant has nothing to consult on except completed code, which §8.5 line 804 explicitly identifies as the wrong leverage point. The honest framing is that rolling consulting is a Phase A item that can only be funded if Phase B-style funding closes before Phase A's design-implementation work substantially completes — i.e., it is a timing-conditional Phase A item, not a Phase B aspiration.
- **RECOMMENDATION:** Rename this item "Phase A consulting aspiration, fundable via early-closing Phase B grant," and acknowledge explicitly that the value of this funding diminishes as Phase A implementation proceeds. Alternatively, reframe Phase B to include "review of completed-but-not-yet-audited cryptographic implementation" as a distinct deliverable from "rolling review during design" — but the brief currently conflates them.

## Significant findings

### F5: Phase D "recurring audit cycle" amortization hides a Phase E gate

- **PROBLEM:** §10.4 line 1106 lists "recurring audit cycle" as a Phase D annual line item, "amortized to roughly $40,000-100,000 per year." But the audit itself is a discrete pre-broader-release-style event 18-24 months after the first audit (§8.5 line 815). Amortizing it into Phase D steady-state cost frames it as ongoing operating expense, but the actual funding requirement is a lump sum of $60K-150K (per §10.3 line 1085, since this is functionally a re-run of the pre-beta full audit at scope-adjusted depth) at a specific gate, not $40-100K/year continuous. The brief lacks a Phase E, but the recurring audit functionally is one.
- **EVIDENCE:** docs/design-brief.md line 815 ("A second external cryptographic audit approximately 18-24 months after the first"); line 1106 (Phase D amortization); line 1085 (pre-beta audit market rates).
- **IMPACT:** A funder evaluating Phase D sustainability sees $90K-250K/year and reads it as smooth recurring expense. The actual cash flow has a $60K-150K bolus at month 18-24. If the project cannot raise that bolus, Phase D's "recurring audit cycle" line collapses — the project would need to defer the second audit and the no-skip-the-audit posture (§8.5 line 813) interacts with this. The phase model does not currently surface the lumpy nature of this expense.
- **RECOMMENDATION:** Either (a) introduce Phase D's recurring audit as a "Phase D event, year 2" line with the full $60-150K figure stated as a discrete gate, or (b) introduce a Phase E label for the second audit cycle and structure ongoing operations as "Phase D between Phase E events." The amortization framing is misleading when the underlying cash flow is bursty.

### F6: Foundation overhead in Phase D presupposes a successful Phase C foundation-incorporation event

- **PROBLEM:** §10.4 line 1108 lists "Foundation overhead: $10,000-30,000 per year" as a Phase D recurring cost. But foundation incorporation is in Phase C (line 1087) and §10.7 line 1160 explicitly contemplates a scenario where "Foundation incorporation funding does not close." In that case, Phase D's "foundation overhead" line does not exist — the natural-person-operated structure continues with different (probably lower or different in kind) overhead. Phase D's $90K-250K range silently assumes Phase C succeeded at the foundation-incorporation component.
- **EVIDENCE:** docs/design-brief.md line 1087 (Phase C foundation incorporation); line 1108 (Phase D foundation overhead); line 1160 (foundation incorporation may not close); line 1112 ("Phase D annual totals without maintainer compensation" presented as a range without scenario flag for foundation-failed case).
- **IMPACT:** Phase D as written is one scenario among several depending on which Phase C components closed. A funder cannot tell whether the $90K floor includes or excludes foundation overhead when the foundation did not incorporate. The phase model promised a clean "what unlocks what" mapping, but Phase D is conditionally defined on Phase C outcomes the brief does not enumerate.
- **RECOMMENDATION:** Add a Phase D scenario branch: "Phase D (foundation-operating)" with the $90-250K range, and "Phase D (natural-person continuing)" with the foundation-overhead line replaced by the residual self-funded posture cost. The §10.7 risk notes the scenario but Phase D's totals do not reflect it.

### F7: UX engineer "Optional in one sense, not in another" muddies its phase commitment

- **PROBLEM:** §10.3 line 1088 describes the UX engineer as "optional in the sense that the developer can complete the work solo at slower cadence; it is not optional in the sense that the work itself is in scope." This straddles Phase C (Phase C funding can hire one) and Phase A (developer-solo completes the work). If Phase C closes only at floor ($105K, line 1093), the UX engineer is not engaged (line 1098) and the work falls back to developer-solo, which means it is back in Phase A — but Phase A's open-ended timeline (line 1024) means it never had a calendar floor for this work.
- **EVIDENCE:** docs/design-brief.md line 1088 (UX engineer optional/not-optional framing); line 1093 (Phase C floor excludes UX engineer); line 1098 (Phase C contingency: UX engineer not engaged); line 1024 (Phase A no calendar commitment).
- **IMPACT:** The §5.6 implementation surface (30-50% of v1 effort per the engineering review) is the largest single uncertainty under the Phase C-floor scenario. If it bounces back to Phase A, broader release is gated on Phase A's volunteer-baseline cadence completing the largest piece of UI work — which the brief never committed to. The funding model should make this explicit: Phase C-floor outcomes mean the UX work re-enters the open-ended Phase A timeline.
- **RECOMMENDATION:** Add to Phase C contingency: "Without UX engineer funding, the UX work returns to Phase A's open-ended cadence; broader release is gated on the developer completing this work at volunteer-baseline pace." Currently the phase contingency hides this in the §10.4 silence about what Phase D-readiness requires.

### F8: Localization/translation honoraria phase entry is ambiguous

- **PROBLEM:** §10.4 line 1110 lists "Localization and translation honoraria" as a Phase D item ($5K-20K/year). But the architectural slot exists from Phase A (line 559: "v1 ships English-only... the architectural slot uses Android string resources from the start so that v1.5 localization additions do not require a code refactor"), and §8.6 line 835 identifies localization as a partnership category active in pilot scope. The Phase D placement implies translation funding doesn't start until steady-state operations, but partner-organization localization conversations (Phase A per §10.1 line 1034) and v1.5 release-cycle localization work would predate Phase D.
- **EVIDENCE:** docs/design-brief.md line 559 (English-only v1, slot from Phase A); line 835 (localization partnership in pilot scope); line 1034 (Phase A includes partner outreach for localization); line 1110 (Phase D localization line).
- **IMPACT:** Smaller than the others, but creates ambiguity for partner organizations evaluating localization participation: under which funding cycle are translators compensated? If a partner asks "when does the project fund translation work for our language," the brief points to Phase D — but the operational reality is that v1.5 release work happens before Phase D's steady state is reached.
- **RECOMMENDATION:** Either move the localization honoraria line to Phase C (where v1.5 release work falls), or split it across Phase C (one-time setup of first translations alongside v1.5 release) and Phase D (ongoing maintenance). Phase D alone is too late for the operational reality §8.6 implies.

## Minor findings

### F9: Phase A "two-to-four GrapheneOS-capable Pixel devices" sizing is unjustified at the boundary

- **PROBLEM:** §10.1 line 1042 commits Phase A to 2-4 Pixel devices for "developer testing and a small loaner pool for partner-facilitated pilot training." Pilot training is Phase B work (post-audit, line 1075). At Phase A, only developer-testing devices (1-2) are required; the loaner-pool devices (2-3) are pre-positioned for an event Phase A is not committed to reach.
- **EVIDENCE:** docs/design-brief.md line 1042 (Phase A 2-4 devices); line 1075 (Phase B unlocks pilot); line 1051 (Phase A "cannot deliver" pilot deployment at scale).
- **IMPACT:** Minor cost leak from Phase B into Phase A. ~$1,000-2,000 of the Phase A absorbed budget is functionally Phase B pre-positioning.
- **RECOMMENDATION:** Either justify the loaner-pool acquisition as Phase A insurance (pre-positioning before grant decision risk), or move 2-3 of the 4 devices to Phase B's "$0-500/year incremental cost" line.

### F10: Phase B "infrastructure operational costs" is described as $0-500/year incremental, but this includes nothing that wasn't already in Phase A's absorbed costs

- **PROBLEM:** §10.2 line 1064 lists "initial infrastructure operational costs for the pilot phase: domain registrar (already in Phase A), email/PGP infrastructure for disclosure handling (free), continued static hosting (free); incremental cost approximately $0-500/year." The entire line is explicitly $0 — it does not deserve a line item because it does not represent funded work.
- **EVIDENCE:** docs/design-brief.md line 1064 (all items free or already-in-Phase-A); line 1068 (Phase B floor includes "ongoing infrastructure" but the floor math doesn't change).
- **IMPACT:** Phase B inventory padding. The line creates the impression Phase B includes infrastructure funding when it does not.
- **RECOMMENDATION:** Remove the line, or move it to a "Phase B does not require additional funding for X" subsection to keep the inventory honest.

### F11: Phase C "pilot deployment hardware: minimal incremental cost" is a Phase A item written into Phase C

- **PROBLEM:** §10.3 line 1089 lists "pilot deployment hardware" as Phase C with "minimal incremental cost." But the BYOD model and the 2-4 device project pool are Phase A (line 1042). Phase C does not need to mention this line item at all — it is Phase A inherited state.
- **EVIDENCE:** docs/design-brief.md line 1089 (Phase C hardware line); line 1042 (Phase A hardware).
- **IMPACT:** Padding; creates an impression that Phase C funds hardware when it does not.
- **RECOMMENDATION:** Remove the line or merge into a "Phase C inherited from Phase A" note for clarity.

## Patterns

Three patterns emerge across the findings:

**Pattern 1: Items with developer-absorption escape clauses.** Phase B legal consultation (F1) and the "developer can complete UX work solo" framing (F7) both create dual-state items where the item exists in one phase if funded and another phase if not. This is a real operational pattern but it should be made first-class rather than buried in escape-clause phrasing. The phase model needs a "conditional-phase" or "developer-absorbed-fallback" category, or each affected item needs explicit dual phase-placement.

**Pattern 2: Sequential dependencies presented as phase-gated funding events.** Foundation incorporation in Phase C (F2) depends on Phase B legal consultation having completed. Phase D foundation overhead (F6) depends on Phase C incorporation having closed. These are dependencies, not parallel gates. A funder evaluating "what does Phase C unlock?" needs to know which Phase B/A prerequisites are assumed. The current phase descriptions present each phase as standalone; the dependency graph is implicit.

**Pattern 3: Smoothed averages hide lumpy gates.** The Phase D "recurring audit cycle" amortization (F5) is the clearest case: a $60-150K event at month 18-24 is presented as $40-100K/year smooth expense. This pattern would also bite if reviewer honoraria (F3) are quietly assumed to begin mid-Phase-B rather than at a clean Phase C transition. The phase model is honest about discrete gates at the macro level; it should preserve discrete-gate honesty at the line-item level rather than smoothing into operating costs.

The cumulative cost of these leaks is that a funder asking the central §10 question — "what does my dollar unlock at each phase?" — cannot get a clean answer for any item with cross-phase dependency. The phase model's structural advantage over a flat budget is exactly that clean-mapping property; the boundary leaks identified above erode it.
