## Summary

The brief contains a schema inconsistency between `docs/architecture-diagrams.md:275` and `docs/design-brief.md:457-465` that names two different "nine-field" envelopes. Either is implementable; both being canonical is not. This is the single highest-confidence finding because an audit firm will ask which version is the spec on day one and pre-implementation reconciliation is free. Below that, the construction has multiple under-specifications that the brief defers to "the implementer will follow RFC X" but RFC X does not pin the choices in question: the `prior_hash` hash function is unnamed (Architecture diagrams 4 and Brief §5.2 both say "hash" without an algorithm); the `issuer_cert_hash` is specified as SHA-256 but the **bytes** it hashes are not defined; the COSE_Sign1 `Sig_structure` (RFC 9052 §4.4) construction — which is what is actually signed, not the CBOR payload — is not specified; and domain separation between the capability-token envelope and the trust-graph-operation envelope is absent despite both using the same operational identity. The 48-hour delay-and-confirm in D0005 is enforced on a fresh device whose clock the brief does not bind to any source — for the Cellebrite-class adversary §3.3 names, this is a circumventable defense. The capability-token authority model is ambiguous between "cryptographic enforcement" and "software permission check"; the brief mixes language consistent with both. None of these findings is fatal; all need to be resolved before code is written rather than after. What the brief does non-trivially well: splitting the Ed25519 seed rather than the derived scalar (preserves RFC 8032 deterministic nonces — a common bug in homemade split-key Ed25519); the withdrawal-vs-compromise revocation split with `revoked_as_of` (closes the cascade-laundering attack the §5 review correctly identified); the commitment-only Sigsum logging pattern (closes the public-metadata-channel surface §3.3 names).

## High-confidence findings

### H1. The nine-field schema is documented two ways; neither is marked canonical

- **PROBLEM:** `docs/architecture-diagrams.md:275` enumerates the nine fields as: "operation type, issuer, subject, prior hash, issuer-cert-hash, scope, timestamp, payload, signature." `docs/design-brief.md:457-465` enumerates them as: "Issuer, Issuer-cert-hash, Subject, Context, Strength, Timestamp, Expiry, Prior-hash, Signature." Six fields overlap; three differ in each direction. The diagrams have `operation type`, `scope`, `payload`; the brief has `Context`, `Strength`, `Expiry`. Both claim "nine fields"; both reference [D0006](decisions/D0006-cryptographic-envelope.md); neither is marked the canonical specification.
- **EVIDENCE:**
  - `docs/architecture-diagrams.md:275` — "operation type, issuer, subject, prior hash, issuer-cert-hash, scope, timestamp, payload, signature"
  - `docs/design-brief.md:457-465` — fields enumerated as Issuer, Issuer-cert-hash, Subject, Context, Strength, Timestamp, Expiry, Prior-hash, Signature
  - `docs/decisions/D0006-cryptographic-envelope.md:73` — refers to "the eight-field schema is now nine-field (or ten depending on the operation type)" with `revocation_kind` / `revoked_as_of` as conditional additional fields — implying the canonical count is itself version-dependent.
- **IMPACT:** Any auditor's first question is "which schema do I review?" Worse: a solo Rust developer reading the diagrams will implement one schema, then read the brief and implement another, then read D0006 and find that `revocation_kind` and `revoked_as_of` may or may not bring the count to ten. Without reconciliation, two clients implementing from different documents will produce non-verifying signatures.
- **RECOMMENDATION:** Designate D0006 (or a new D0006a appendix) as the canonical schema spec. Replace both lists with a verbatim reference. The canonical schema must enumerate: (1) every field name; (2) its CBOR major type and encoding rule; (3) whether it is mandatory, optional, or operation-type-conditional; (4) the COSE protected/unprotected/payload placement; (5) whether it participates in the signed bytes. Do not ship "nine fields" as a count if the actual count varies by operation type — write `Signed_Operation { common_fields: [...]; operation_type_specific_fields: [...] }` explicitly.

### H2. The `prior_hash` hash function is not specified; the chain scope is documented inconsistently

- **PROBLEM:** Neither D0006, the brief, nor the architecture diagrams name the hash function used for `prior_hash`. The architecture diagrams describe the chain as "an append-only sequence per issuer" (`docs/architecture-diagrams.md:275`); the brief describes it as "a per-(issuer, subject) chain" (`docs/design-brief.md:464`). These are different chain shapes with different equivocation-detection properties.
- **EVIDENCE:**
  - `docs/architecture-diagrams.md:275` — "The prior-hash chain creates an append-only sequence per issuer"
  - `docs/design-brief.md:464` — "Prior-hash — a hash of the issuer's most recent prior operation against the same subject, forming a per-(issuer, subject) chain"
  - `docs/decisions/D0006-cryptographic-envelope.md` — does not specify the hash function for the chain. Only `issuer_cert_hash` is specified as SHA-256 (line 22).
  - `docs/design-brief.md:473` — "The per-(issuer, subject) prior-hash chain is computable from the operation commitments without revealing operation content" — confirms the per-pair scope. The brief and D0006 align; the diagrams contradict.
- **IMPACT:** Per-issuer single chain detects ALL issuer equivocation across all subjects (stronger); per-(issuer, subject) chain only detects equivocation within a (issuer, subject) pair (weaker). An attacker who controls the issuer's signing key can drop an entire (issuer, subject) chain undetectably under the per-pair scheme — they simply omit the chain entries they want to suppress. No verifier examining a different (issuer, subject') chain has any link to the dropped chain. Beyond the scope inconsistency: an unspecified hash function leaves cryptographic agility unresolved (SHA-256 vs SHA-512 vs BLAKE2b vs BLAKE3 produce different bytes and different signature inputs). Two implementations using different hash functions cannot verify each other's chains.
- **RECOMMENDATION:** (1) Specify the hash function explicitly in D0006: SHA-256 is the obvious default since the brief already uses it for `issuer_cert_hash` and the COSE algorithm vocabulary (RFC 9053) names SHA-256 as the standard hash. State this in D0006 with the byte-input definition: "`prior_hash := SHA-256( canonical_cbor_encoding( prior_operation_signed_bytes ) )`" — specify exactly which bytes are hashed. (2) Reconcile the chain scope: keep per-(issuer, subject) (which is what §5.2 commits to) and correct the architecture diagram. Document the security tradeoff explicitly — per-pair scope means the chain detects only same-subject equivocation; cross-subject equivocation detection depends on external auditors examining the Sigsum commitments globally.

### H3. The COSE_Sign1 `Sig_structure` (what actually gets signed) is not specified

- **PROBLEM:** RFC 9052 §4.4 defines the synthetic structure that COSE_Sign1 signatures cover: a CBOR array `[ context: "Signature1", body_protected: bstr, external_aad: bstr, payload: bstr ]`. The signature is over the deterministic encoding of this synthetic structure, not over the payload alone. The brief and D0006 specify "deterministic CBOR encoding (RFC 8949 §4.2.1)" for the **payload** but do not address: (a) whether the protected header is also deterministic; (b) what `external_aad` value is used; (c) whether the `coset` crate produces the same Sig_structure bytes that an independent COSE implementation (e.g., `cose-rust`, RustCrypto's COSE work, Python `cose`) would produce.
- **EVIDENCE:**
  - `docs/decisions/D0006-cryptographic-envelope.md:49` — "Signed content uses deterministic CBOR encoding (RFC 8949 §4.2.1) for signature reproducibility."
  - `docs/design-brief.md:418` — "the payload encoded as deterministic CBOR (RFC 8949 §4.2.1) so two implementations produce byte-identical signed content"
  - `docs/design-brief.md:455` — "deterministic-CBOR-encoded payload (RFC 8949 §4.2.1) so byte-identical signed content is reproducible across implementations"
  - Neither D0006 nor the brief mentions RFC 9052 §4.4, `Sig_structure`, or `external_aad`. The deterministic-CBOR claim is asserted of the payload, not of the synthetic signed structure.
- **IMPACT:** This is the canonical CBOR-canonicalization-mismatch attack surface from the cryptographer prompt's adversarial agenda. Two implementations could each correctly produce deterministic CBOR for the payload, then produce different Sig_structure bytes because (a) they encode the protected header differently (different deterministic-encoding profile, or non-empty `external_aad`); (b) they handle the protected-header empty-map / absent-protected-header case differently (RFC 9052 §3 specifies the encoding as `bstr` even when the underlying map is empty — implementations get this wrong). Result: clients signing with one library cannot be verified by clients using another, breaking the v1.5 multi-implementer reviewer-pool ideal.
- **RECOMMENDATION:** D0006 must specify (1) the exact `Sig_structure` form including the `context: "Signature1"` constant and the empty-`bstr` external_aad; (2) that the protected-header bytes are produced by deterministic CBOR encoding using RFC 8949 §4.2.1 — and explicitly include the protected header in the determinism requirement, not just the payload; (3) the specific behavior of the `coset` crate's `CoseSign1::tagged_value` vs `to_vec` paths, since one is COSE-tagged and the other is not — pick one and require it. (4) Add a test vector to D0006 (a concrete (key, payload, expected bytes, expected signature) triple) so the second-implementer constraint is verifiable. Without a test vector, every future implementer reinvents the canonicalization decisions privately.

### H4. The `issuer_cert_hash` binding is specified as SHA-256-of-something; the something is not defined

- **PROBLEM:** D0006:22 specifies `issuer_cert_hash` as "a SHA-256 hash of the master attestation certifying the operational key being used to sign." It does not specify what bytes constitute "the master attestation." Candidates: (a) the COSE_Sign1 outer-encoded bytes; (b) the CBOR payload bytes; (c) the Sig_structure bytes; (d) the master signature bytes only; (e) some canonical "claim set" the brief has not defined. Each produces a different hash.
- **EVIDENCE:**
  - `docs/decisions/D0006-cryptographic-envelope.md:22` — "addition of `issuer_cert_hash`: a SHA-256 hash of the master attestation certifying the operational key being used to sign."
  - `docs/design-brief.md:458` — "a SHA-256 hash of the master attestation certifying the issuer's operational key" — same wording, no further specification.
- **IMPACT:** Without byte-level specification, two implementations will produce different `issuer_cert_hash` values for the same logical master attestation. Operations signed by client A will not verify under client B's reconstruction of the master-attestation-hash chain. This is the same class of bug as H3 (canonicalization drift) but applied to a chain-binding hash; its symptom is "your master attestation does not verify mine" rather than "your signature does not verify."
- **RECOMMENDATION:** Specify in D0006: `issuer_cert_hash := SHA-256( deterministic_cbor_encoding( master_attestation.Sig_structure ) )`. Then either: (a) bind it to the canonical operation envelope (the COSE_Sign1's tagged bytes) which is what most attesters intuitively expect; or (b) bind it to the master attestation's `Sig_structure` (which is what the signature itself covers — the most defensible choice from a "signed bytes are the canonical bytes" standpoint). Either choice is defensible; the absence of a choice is not. Add a test vector.

### H5. The 48-hour delay-and-confirm is enforced on a fresh device whose clock the brief does not bind

- **PROBLEM:** D0005:27 specifies that after shares arrive, "share release is held for a mandatory 48-hour cooling-off period before the master is reconstructed." `docs/architecture-diagrams.md:309-333` shows shares are released by peers to the fresh device, then the fresh device begins a 48-hour countdown locally before reconstructing the master. The fresh device is, by definition, a newly-acquired device whose system clock the user has just configured. The brief specifies no monotonic-source binding, no TLS-time check, no peer-side time-witnessing, no Sigsum-timestamp anchor. A Cellebrite-class adversary holding the device per §3.3 (Returned-after-seizure surface) — or a software adversary in the device's OS — can advance the system clock to compress the 48-hour window to seconds.
- **EVIDENCE:**
  - `docs/decisions/D0005-peer-verification.md:27` — "After all required shares have arrived and verification challenges have passed, share release is held for a mandatory **48-hour cooling-off period** before the master is reconstructed."
  - `docs/decisions/D0005-peer-verification.md:30` — "The fresh device displays the time remaining and the channels through which the user can cancel."
  - `docs/architecture-diagrams.md:319-333` — sequence diagram shows shares received → "Recovery confirmed pending; 48-hour window begins" → "48 hours elapse" → reconstruct.
  - No reference to NTP, monotonic clock, TLS handshake timestamp, Sigsum timestamp, or peer-time-witness anywhere in D0005, D0006, the architecture diagrams, or the brief's §5.3.
- **IMPACT:** Attack sketch: (1) Adversary obtains user device (post-seizure scenario or device returned with implant); (2) adversary obtains peer challenges (per §3.3 Recovery network surface: via HUMINT, coercion, or peer device compromise); (3) adversary installs fresh device, initiates recovery, satisfies peer challenges, receives shares; (4) adversary advances device clock 48 hours; (5) device performs reconstruction. The 48-hour window — the only mechanism §5.3 names as resisting "a determined adversary who controls the user for that long" — fails closed-loop within minutes. The 90-day stale-flag escalation timer in D0006:40 has the same clock-source ambiguity, with similar implications.
- **RECOMMENDATION:** Two options, both implementable: (a) **Peer-side enforcement.** Peers hold their shares until 48 hours after challenge verification, then release. The clock is on the peer device, not the recovering device. The recovering device's clock cannot influence the timing. This is the architecturally correct choice; it requires the architecture diagram to be updated (shares are released AFTER 48h, not before). (b) **Sigsum-anchored time.** The recovery commitment is logged to Sigsum at initiation; reconstruction requires fetching a Sigsum-witnessed log entry timestamped at-least-48h-after-the-commitment. Sigsum entries are witnessed and tamper-evident; the recovering device cannot fabricate the time gap. This option preserves the existing share-release-then-wait flow but ties the wait to an external witness. (a) is simpler and matches the intuitive read of "the user can cancel within 48 hours by contacting peers." Choose (a).

### H6. The capability-token authority model is cryptographically ambiguous

- **PROBLEM:** The brief describes capability tokens in two registers that imply different security properties. Register 1 (cryptographic enforcement): tokens "authorize specific device operations" (`docs/architecture-diagrams.md:211`) and a device "cannot, by holding its current token, exceed the permissions that token explicitly granted" (`docs/design-brief.md:422`). Register 2 (software permission): "Device tokens. Standard application storage, encrypted at rest by the device's full-disk-encryption key. Token material is recoverable from device storage if the device is unlocked" (`docs/design-brief.md:438`); operations are signed by "the operational identity (Section 5.1), so a device whose operational key has been revoked cannot continue to issue graph operations" (`docs/design-brief.md:495`). If operations are signed by the operational identity (which lives in hardware-element + passphrase gating), and the device holding the token has access to the operational identity's signing oracle, then the capability token is a software-level permission check, not a cryptographic constraint. The brief does not specify whether (a) device key signs operations alongside the operational identity's signature, with capability tokens proving the device-key-to-operational-key delegation; or (b) capability tokens are software policy advice with no cryptographic enforcement beyond the operational identity itself.
- **EVIDENCE:**
  - `docs/design-brief.md:418` — "Each token's claim set names an issuer (the operational identity public key), a subject (the device key being delegated to), a set of scope strings..., and the operational identity's signature over all the above."
  - `docs/design-brief.md:438` — "Device tokens. Standard application storage, encrypted at rest by the device's full-disk-encryption key."
  - `docs/design-brief.md:422` — "a device with a `messaging-send` token cannot issue trust graph attestations even if its key material is fully extracted by forensic tooling."
  - `docs/design-brief.md:495` — "Signing of new operations uses the operational identity (Section 5.1)"
  - No specification of whether the device key (the token's "subject") signs anything itself, or whether the device-key-to-operational-identity proof-of-possession is required for operations.
- **IMPACT:** If capability tokens are software-only enforcement (Register 2), then `docs/design-brief.md:422`'s claim — that forensic extraction of one device's key material cannot exceed its token's scope — is false. A forensic extractor who pulls the operational identity's hardware-gated signing oracle credential (via implant or coerced passphrase) can sign anything the operational identity can sign, regardless of any device's token. The token does nothing. If capability tokens are cryptographic (Register 1), then operations must be co-signed by the device key, and the brief must specify the multi-signature envelope, the device-key generation/storage model, and the device-key-revocation flow — none of which are in §5.1 or D0006.
- **RECOMMENDATION:** Pick one. **Recommended: Register 1 (cryptographic).** Specify in D0006 or a new D0006b: (1) Operations are signed by the **device key**, not directly by the operational identity. (2) The device's signature is verified against the device key declared as the `subject` of a current capability token. (3) The capability token (signed by operational identity) is included or referenced in the operation envelope so verifiers can chain device-key-signature → capability-token → operational-identity-signature → master-attestation. (4) The device key is generated in hardware where available (StrongBox) and is its own hardware-element-bound credential separately from the operational identity. This makes `docs/design-brief.md:422`'s scope-bounding claim true. If Register 2 is chosen, delete `docs/design-brief.md:422` — the claim it makes is not delivered by the construction.

## Medium-confidence findings

### M1. Cross-protocol signature confusion between capability tokens and trust-graph operations

- **PROBLEM:** Capability tokens and trust-graph operations both use COSE_Sign1, both are signed by the operational identity, both use deterministic CBOR. Without explicit domain separation, the same Ed25519 signing oracle produces signatures for both. An adversary who can get the operational identity to sign a payload "as a capability token" could potentially re-interpret those signed bytes as a trust-graph operation, or vice versa, depending on how the payload structure overlaps.
- **EVIDENCE:**
  - `docs/design-brief.md:418` — capability tokens use `COSE_Sign1` with deterministic-CBOR-encoded payload.
  - `docs/design-brief.md:455` — trust-graph operations use the same `COSE_Sign1` envelope with the same deterministic-CBOR-encoded payload.
  - Neither D0006 nor the brief specifies a domain-separation tag, distinct algorithm parameter, or distinct `external_aad`.
- **IMPACT:** The COSE_Sign1 protected header's `alg` is the same (Ed25519) for both. The `type` field (if it exists in the schema — H1 leaves this open) would be in the payload. If a CBOR map's field ordering or absence handling differs between client A's capability-token verifier and client B's trust-graph-operation verifier, a payload that decodes as one in one client might decode as the other elsewhere. The attack is hypothetical until the schema is locked, but the absence of domain separation is a class-of-attack the COSE working group designed around with the `Sig_structure` and external_aad mechanisms.
- **RECOMMENDATION:** Use the COSE_Sign1 `external_aad` field as a domain tag: `external_aad := "cairn-v1-capability-token"` for capability tokens, `external_aad := "cairn-v1-trust-graph-operation"` for trust-graph operations. Specify this in D0006. The Sig_structure binds to the external_aad; a signature produced for one domain cannot verify in the other even with payload-bit overlap. This is the standard COSE pattern for cross-protocol separation.

### M2. Pre-shared peer challenge replay protection is unspecified

- **PROBLEM:** D0005 specifies that at provisioning the user and each peer establish a shared phrase, and that at recovery the peer challenges the user to produce the phrase. The brief does not address whether the challenge value is single-use or reusable across recovery attempts. If the peer asks the same phrase on every attempt, an adversary who observes one successful recovery (via shoulder surfing, channel compromise, or HUMINT) has the phrase indefinitely.
- **EVIDENCE:**
  - `docs/decisions/D0005-peer-verification.md:18-23` — describes the phrase establishment and verification flow without specifying single-use or counter semantics.
  - `docs/design-brief.md:505` — "The phrases never travel through the recovery-request channel; they exist for the peer to challenge the user with at share-release time." No mention of phrase rotation, counter, or salt.
- **IMPACT:** Recovery is a low-frequency event but not necessarily a one-time event (peer rotation, suspected compromise, voluntary key rotation). A user who recovers once and is later targeted again gets no fresh phrase. The phrase becomes a long-term static credential — closer to a password than to a cryptographic nonce. For the §3.3 Recovery network surface — where peers themselves may be compromised — this means a compromised peer who saw one recovery has the phrase forever.
- **RECOMMENDATION:** Specify in D0005 that phrases are single-use: each successful peer challenge results in a NEW phrase agreed (out-of-band) for the next recovery attempt. Alternatively, specify a phrase-plus-counter scheme where the user produces `phrase || counter_n` for the n-th challenge, with the counter cryptographically bound to the share content. The former is simpler operationally and matches the threat model better; the latter is closer to standard challenge-response design. Either is implementable; the absence of specification is what produces the static-credential pattern.

### M3. Sigsum split-view detection thresholds and witness threshold semantics are unspecified

- **PROBLEM:** The brief commits to "Sigsum-anchored release log with witness cosignatures" (`docs/decisions/D0015-v1-release-security-posture.md:33`) and "witness cosignatures via the same witness pool that anchors the trust graph" but does not specify: (a) how many witnesses the client requires to consider a Sigsum entry valid; (b) what failure mode triggers when fewer-than-required witnesses cosign; (c) what the threshold-vs-strict-majority semantics are; (d) whether different operation types (trust-graph commitments, release logs) use the same or different thresholds.
- **EVIDENCE:**
  - `docs/decisions/D0015-v1-release-security-posture.md:33` — "Witness cosignatures via the same witness pool that anchors the trust graph; witness-pool recruitment conditional on Q5 (NGO partner outreach) per §8.6"
  - `docs/design-brief.md:597` — "Witnesses — recruited NGO and academic partners (Q5; witness pool is a v1 commitment per §8.6) — cosign the release log state, so log tampering by the log operator is itself detectable."
  - `docs/open-questions.md:Q10` — "Specific organizations and individuals not yet identified for either pool." No threshold parameters specified.
- **IMPACT:** Sigsum's whole-system security claim against split-view attacks rests on the witness count and threshold. Sigsum itself (the protocol) supports witness cosigning but does not fix the threshold — the deployer chooses. Without a specified threshold, the client cannot meaningfully reject a Sigsum entry as "insufficiently witnessed." A single-witness deployment provides no split-view protection (the witness and the log operator can collude); a threshold-of-1-of-many is functionally equivalent. The brief commits to recruitment work but not to the protocol threshold the recruitment must satisfy.
- **RECOMMENDATION:** Specify in D0015 (or a new D0015a): (1) Minimum witness count for v1 (e.g., 3 witnesses); (2) Threshold for acceptance (e.g., 2-of-3 cosignatures required); (3) Failure mode (client rejects the Sigsum entry and the operation it commits to). State this as an architectural property like the 3-of-5 Shamir threshold, not as an operational policy. The recruitment pool size in Q10 must satisfy the threshold; the threshold drives the recruitment ask, not the other way around.

### M4. Master re-split atomicity under interruption is unspecified

- **PROBLEM:** The recovery flow in `docs/architecture-diagrams.md:326-333` shows: reconstruct master → generate new operational identity → sign with master → re-split master → zeroize master → log key rotation. The flow does not specify what happens if the fresh device is seized, crashes, or loses network between the reconstruct step and the re-split step (e.g., after 2 of 5 peers receive new shares but 3 do not).
- **EVIDENCE:**
  - `docs/architecture-diagrams.md:326-333` — sequence diagram steps without persistence-failure-mode specification
  - `docs/design-brief.md:511` — "The reconstructed seed is then immediately re-split using Shamir among the peer set... and erased from active memory." No discussion of partial-failure handling.
- **IMPACT:** Three scenarios this leaves underspecified: (a) Mid-re-split crash with master still in memory — does the device restart resume re-split, or is the master gone but partial new shares are out? (b) Mid-re-split seizure — does the adversary find the master in memory if the zeroize step hasn't fired? (c) Network failure during re-split — do peers receive partial shares that, combined with old shares, reduce the effective threshold? The §3.3 Returned-after-seizure surface and the master-in-memory-during-recovery window in §5.1 both contemplate the recovery moment as the master's exposure window; the partial-failure modes during that window are where forensic adversaries operate.
- **RECOMMENDATION:** Specify in D0005 (or a new D0005a): (1) The recovery flow is atomic with respect to the master's lifetime — all peers receive new shares before any zeroize step; the master is held in `secrecy`-wrapped pinned memory across the full operation; the flow either completes in full or no new shares are distributed. (2) If new shares cannot all be delivered, the master is zeroized, the new-share fragments are discarded by recipient peers, and the recovery is treated as failed-but-non-leaking. (3) The new operational identity is signed by the master only after the re-split has succeeded, not before — so a partial-completion does not produce a "user has new operational identity signed by master, but the master was leaked because re-split partially completed and then crashed." Document the cryptographic ordering explicitly.

### M5. Equivocation detection requires cross-user audit; the brief does not specify who audits

- **PROBLEM:** The per-(issuer, subject) prior-hash chain detects equivocation only when the same observer sees both branches of the equivocation. If issuer I attests to subject S twice with different content but the same prior_hash, that's an equivocation — but it's only detectable if some observer sees both attestations. User A sees one via SimpleX channel A-I; user B sees the other via SimpleX channel B-I. Both submit hashes to Sigsum (commitment-only). The hashes do not reveal that the operations conflict. Detection requires someone with the operation content (not just the commitment) to compare two operations submitted under the same chain.
- **EVIDENCE:**
  - `docs/design-brief.md:464` — "Prior-hash — a hash of the issuer's most recent prior operation against the same subject, forming a per-(issuer, subject) chain that lets receivers detect equivocation."
  - `docs/design-brief.md:473` — "The per-(issuer, subject) prior-hash chain is computable from the operation commitments without revealing operation content, so external auditors detect equivocation patterns without participating in any specific user's trust computation."
  - The "external auditors" are not specified; no monitoring infrastructure, no detection mechanism, no notification flow is described.
- **IMPACT:** The brief claims equivocation detection as a v1 property of the trust-graph design, but the detection mechanism is "external auditors" who are not named, funded, or operationally specified. In practice: there is no equivocation detection in v1 unless someone independently audits the Sigsum log against the SimpleX-propagated operations — which requires access to operations the brief specifies as never publicly logged. The detection is structurally possible but operationally unimplemented.
- **RECOMMENDATION:** Either (a) commit to a specific audit operator (the project itself, a named partner, an academic group) with a documented audit cadence and notification path; (b) honestly downgrade the claim: "the trust graph design supports equivocation detection by any party with access to both branches of an equivocation; v1 does not commit to systematic detection infrastructure"; (c) ship a per-client lightweight detector — every client that receives an operation submits the operation hash to Sigsum and also publishes (to a dedicated channel) the commitment with a flag if the prior_hash chain diverges from what the client previously held. (c) is the cryptographically interesting option; (b) is the honest fallback.

### M6. The reconstruction-window forensic-extraction claim is bounded by what zeroize cannot guarantee

- **PROBLEM:** D0003:43 and §5.1 claim that the `zeroize` crate guarantees secret-material destruction on scope exit, and that this defends against the §3.3 Returned-after-seizure surface during the reconstruction window. The Rust `zeroize` crate produces a `volatile_write` of zeros at Drop, but cannot guarantee: (a) that the compiler does not produce a copy elsewhere (some optimizations are subtle, and the volatile_write is on one address only); (b) that the OS has not paged the memory to disk before the Drop ran (pinned memory mitigates this but mlock() can fail silently on Android); (c) that hardware-level cache lines containing the secret have been flushed (zeroize cannot reach those); (d) that a forensic implant resident in the OS has not already exfiltrated the secret during the reconstruction window.
- **EVIDENCE:**
  - `docs/decisions/D0003-implementation-language.md:43` — "The [`zeroize`](https://crates.io/crates/zeroize) crate guarantees secret-material destruction on `Drop`."
  - `docs/design-brief.md:406` — "the `zeroize` / `secrecy` crates enforce destruction on scope exit; pinned memory prevents swap."
  - The architecture diagram acknowledges "Bounded-window exposure (per §5.1 honesty)" but does not name the residual classes (a)-(d).
- **IMPACT:** This is the cryptographer-prompt's "Cellebrite-class adversary against the three-tier identity model" question. The brief acknowledges the reconstruction window as a residual exposure, but the language ("`zeroize` ensures destruction") overstates what `zeroize` actually delivers. A Cellebrite-class adversary with kernel access (e.g., via a returned-after-seizure implant per §3.3) can read the master out of process memory between reconstruction and the next instruction; `zeroize` cannot help here. The defense is bounding the reconstruction window in _time_ (sub-second), not in _space_ (the secret was zeroed). The brief mixes these.
- **RECOMMENDATION:** Tighten the brief's language: replace "`zeroize` ensures destruction" with "`zeroize` ensures the application memory location is overwritten on scope exit, bounding the post-scope exposure window; pre-scope-exit exposure (kernel access, hardware cache, forensic implant resident before scope) is not mitigated by `zeroize` and is named as residual surface." This is the honest framing. The architectural answer to (a)-(d) is "do not perform recovery on a device suspected of compromise" — which the brief states elsewhere but should connect to the specific `zeroize` limitations here.

## Low-confidence concerns

### L1. The `coset` crate's deterministic CBOR conformance is unverified

- **CONCERN:** The brief commits to the `coset` crate as the reference COSE implementation. I have not read the `coset` source. The crate's deterministic-encoding conformance to RFC 8949 §4.2.1 for all field types Cairn uses (Ed25519 alg parameter, byte strings, integer timestamps, optional fields, nested maps in the payload) is not verified by the brief. Some Rust CBOR libraries (including dependencies of `coset`) have historically had subtle non-determinism in map ordering or integer width selection.
- **EVIDENCE:** Speculation. I did not read `coset`. The brief asserts deterministic CBOR conformance but does not cite test vectors or version-pin guidance.
- **IMPACT:** If `coset` does not produce deterministic encoding for all field types, signatures generated by Cairn-via-coset will not match signatures generated by an independent COSE implementation — defeating the v1.5 multi-reviewer binary-equivalence ideal at the cryptographic-protocol level.
- **RECOMMENDATION:** Pre-implementation: write a test that round-trips a known operation through `coset`, compares the produced bytes against a hand-constructed deterministic CBOR encoding, and confirms byte-equivalence. Make this test a CI gate. If the test fails, the choice is either: (a) submit a fix upstream to `coset`; (b) fork `coset`; (c) bypass `coset` for the CBOR encoding step and use a lower-level CBOR library (e.g., `ciborium`) configured for strict deterministic mode.

### L2. Algorithm agility in COSE is partial — the protected-header `alg` is signed

- **CONCERN:** The brief claims "supporting future PQ migration without schema change per Section 3.4 PQ note" (`docs/design-brief.md:418`). The COSE_Sign1 protected header's `alg` parameter is in the SIGNED bytes (it is part of the Sig_structure). An old client receiving an operation with `alg = ML-DSA-65` cannot verify it because it does not know how to perform ML-DSA verification. Forward-compatibility is partial: old clients reject new alg unverifiably, but new clients can still verify old alg.
- **EVIDENCE:** RFC 9052 §4.4 specifies `alg` placement in the protected header; deterministic CBOR encoding of the protected header is therefore part of the signed bytes. `docs/design-brief.md:418` makes the PQ-migration claim but does not address verifier-side compatibility.
- **IMPACT:** This is structurally honest — old clients SHOULD reject signatures they cannot verify. But the brief's framing ("without schema change") implies more than the construction delivers. A migration to PQ requires every active client to update before PQ-signed operations propagate; otherwise PQ-signed operations are silently ignored by the old clients' trust graph view.
- **RECOMMENDATION:** Adjust the brief's language: "The COSE protected-header `alg` parameter allows future migration to alternative signature algorithms (including PQ); migration requires coordinated client update because old clients cannot verify operations signed under unknown alg parameters. The architectural slot is present; the operational migration is not free." This is the honest framing.

### L3. The `vsss-rs` library choice is deferred to Q8 but bounds the pre-pilot audit's "well-understood primitive" claim

- **CONCERN:** D0011 removes the Shamir SSS primitive from the pre-pilot external audit on the grounds that "the GF(256) byte-wise threshold scheme is a well-understood primitive with extensively-audited reference implementations (vsss-rs candidate per Q8)." The `vsss-rs` candidate is not externally audited that I can verify; "extensively-audited reference implementations" gestures at Shamir-in-general rather than `vsss-rs` specifically.
- **EVIDENCE:** `docs/decisions/D0011-audit-budget-and-timing.md:27` — "the GF(256) byte-wise threshold scheme is a well-understood primitive with extensively-audited reference implementations (vsss-rs candidate per Q8)"; `docs/open-questions.md:Q8` — "Specific Shamir Secret Sharing implementation (`vsss-rs` is a candidate; SLIP-39 adaptation is the alternative if a standard share format is preferred)."
- **IMPACT:** If `vsss-rs` (or whatever crate is selected) has a known timing side channel or a known bug in low-threshold edge cases, the pre-pilot audit does not catch it because Shamir is out of scope. The defense is "property tests + source review" — but a property test cannot detect a constant-time violation, and a source review by a non-cryptographer cannot reliably catch subtle GF(256) implementation bugs.
- **RECOMMENDATION:** Verify the chosen Shamir library has either (a) an external audit attesting timing-safety or (b) a documented constant-time implementation strategy that can be verified by a property test (e.g., `cargo bench` showing flat timing across share inputs). If neither, add Shamir back to pre-pilot audit scope — the budget is conserved because the audit attention concentrates on the COSE envelope, but adding "timing-safety verification of the chosen Shamir implementation" is small marginal scope.

### L4. Push-notification metadata is acknowledged but not architecturally bounded

- **CONCERN:** §3.3 names a push metadata surface; v1 defaults push OFF and uses polling at 15 min intervals. But the brief notes that an opt-in to push is possible. The metadata implications of polling at 15-min intervals are not explored — polling traffic is itself observable, the traffic shape may reveal "user is at home" vs "user is traveling" via timing patterns. This is not a cryptographic finding per se but is the kind of thing the §3.3 Metadata surface should consider.
- **EVIDENCE:** `docs/design-brief.md:559` — "v1 ships with push notifications off by default. The client polls at user-configurable intervals (default 15 minutes), accepting the battery and latency cost in exchange for not adding a third-party metadata channel."
- **IMPACT:** Speculative. Polling is observable; whether the observation produces actionable metadata depends on Tor-circuit-level traffic analysis, which the brief generally defers to Tor's threat model.
- **RECOMMENDATION:** Add a short paragraph to §5.4 acknowledging polling-pattern observability and naming this as covered by Tor's threat model rather than by Cairn's. Not a cryptographic-protocol change.

## What works well

Each item below explains the failure mode the design avoids that a naive implementation would not.

**Splitting the 32-byte Ed25519 seed rather than the derived signing scalar.** D0006:48 and design-brief:404 explicitly specify the 32-byte RFC 8032 §5.1.5 seed as the Shamir input. A common bug in homemade split-key Ed25519 systems is splitting the derived scalar — which appears equivalent but breaks the deterministic-nonce contract that RFC 8032 mandates. Reconstructing a scalar instead of a seed and then signing produces deterministic nonces derived from a different input than the master signature would, which is not just an inefficiency but a potential nonce-collision risk if multiple devices ever reconstruct independently. The brief gets this right and explicitly cites the rationale (`docs/decisions/D0006-cryptographic-envelope.md:48` — "Splitting the seed (not the derived scalar) preserves Ed25519's deterministic nonce contract"), which suggests the design team has read the relevant Ed25519 implementation literature.

**The withdrawal-vs-compromise revocation split with `revoked_as_of`.** D0006:31-43 and design-brief:447-449 specify two revocation operation types with different cascade semantics. A naive implementation would have a single "revoke" operation, which forces the cascade response to be one-size-fits-all: either too aggressive (revocation cascades hard-suspend everything downstream, producing recursive collapse) or too soft (everything downstream is just flagged, letting an adversary launder compromised attestations through user re-attestation). The split, plus the `revoked_as_of` timestamp that hard-suspends post-compromise attestations and soft-flags prior ones, is the right shape to close the laundering attack the §5 review (F4) correctly identified. The decision to make re-attestation explicitly non-one-click after key-compromise revocation (`docs/decisions/D0006-cryptographic-envelope.md:39`) addresses the residual user-behavior risk that the cascade design alone cannot close.

**Commitment-only Sigsum logging.** design-brief:473 specifies that operations themselves propagate via SimpleX user-to-user; only hashes (commitments) are submitted to Sigsum. A naive implementation would submit full operations to the transparency log, which makes the log a public metadata channel — issuer/subject/context become publicly enumerable for any observer with log access. The commitment-only pattern preserves Sigsum's auditability (the log is a tamper-evident commit to operation existence and order) while removing the public-metadata leak. The architectural cost (operations live in participant local stores; offline trust-evaluation is harder) is accepted explicitly. This is a well-considered tradeoff and the kind of choice that distinguishes a cryptographically literate design from a checklist design.

**Forward-secrecy / post-compromise-security / at-rest-persistence framing.** design-brief:368-370 and D0006:50 distinguish on-wire FS, on-wire PCS, and at-rest persistence as separable properties. Many secure-messaging briefs conflate "forward secrecy" with "post-compromise security" or quietly elide the at-rest persistence question. Cairn's explicit naming of all three lets users (and reviewers) understand what the architecture protects — past traffic cannot be decrypted on key compromise (FS), future traffic self-heals after compromise (PCS), but at-rest message history is decryptable under unlock (acknowledged residual). This is honesty that matches the §3.5 threat-model framing.

**The decision to use Rust for secret-material lifetime control rather than Kotlin for ergonomics.** D0003 is, on balance, a security decision the brief defends well — the JVM's garbage collector makes secret-material destruction unreliable, and `zeroize`/`secrecy`/`subtle` collectively provide tooling no equivalent exists for in Kotlin. The brief acknowledges the engineering cost of a two-language stack (UniFFI maintenance, bilingual context) honestly rather than pretending Rust is free. The cryptographically correct choice is also the more operationally expensive one here; the brief picks correctness.
