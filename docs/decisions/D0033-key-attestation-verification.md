# D0033 — device-key attestation verification (Android Key Attestation)

**Status:** Accepted (Stage 1)
**Date:** 2026-06-04

> **Confirmed (2026-06-04):** Stage 1 (real `setAttestationChallenge` + the Rust
> verifier to the pinned Google Hardware Attestation Root + the KeyMint-extension
> checks + the My-Identity surface), **advisory** posture (a failed/absent
> attestation downgrades to "unattested" but never bricks the app), accepting the
> key-regeneration migration (fresh installs only). GrapheneOS verified-boot
> pinning + revocation are Stage 2.

> Realizes the D0020 §3.8–§3.9 follow-up flagged in `KeystoreEd25519Signer`
> (`securityLevelOf` comment) and `cairn-uniffi/src/hardware.rs`
> (`AttestationCertificate` doc): cryptographically PROVE the device signing key
> lives in secure hardware (TEE/StrongBox), is non-exportable, and was
> generated in-hardware — by verifying its Android Key Attestation certificate
> chain in Rust, rather than trusting the Kotlin `KeyInfo.securityLevel`
> introspection (which is best-effort + self-reported by the same platform).

## Context

`KeystoreEd25519Signer` (D0028) generates a TEE-backed Ed25519 device key and
reports a `securityLevel` string via `KeyInfo` introspection. That string is a
**self-report**: the platform tells us where it claims the key lives, with no
cryptographic proof. The honest comment at `securityLevelOf` already says the
real proof is the **attestation chain, verified Rust-side**.

Two gaps today:

1. **No real attestation chain is produced.** `generateKeyPair` omits
   `setAttestationChallenge(...)`, so `getCertificateChain` returns only a
   self-signed leaf — no KeyMint attestation extension, no chain to Google's
   root. `attestation_chain` therefore returns an unverifiable single cert.
2. **No verifier.** `cairn-uniffi/src/hardware.rs` declares the
   `AttestationCertificate` type + the `attestation_chain` callback, but nothing
   verifies the chain.

This is downstream of D0020 §3.4 (the hardware-key mediation), §3.8 (Rust-side
attestation verification), §3.9 (GrapheneOS verified-boot binding), and D0028
(the landed `KeystoreEd25519Signer`). The X.509 chain-verification primitive
already exists in-tree: `cairn-sigstore-verify`'s Fulcio path uses
`x509-parser` (the `verify`/`ring` features, D0024 §6.5), reusable here.

## What attestation proves (and what it doesn't)

Android Key Attestation issues, at key-generation time, an X.509 cert **chain**:
`leaf (the device key's cert, carrying the KeyMint attestation extension)` →
`intermediate(s)` → `Google Hardware Attestation Root`. The leaf's extension
(OID `1.3.6.1.4.1.11129.2.1.17`, a `KeyDescription` SEQUENCE) is **signed by the
secure element's attestation key**, which chains to Google's root. So a verified
chain proves: the key was generated by genuine Google-attested KeyMint hardware,
with the attested properties (security level, origin, purpose, the challenge).

It does NOT prove the key is unseizable or that the OS is uncompromised — only
that the _key material_ is hardware-bound + non-exportable as attested. The
GrapheneOS verified-boot binding (§3 below) extends it to "this is GrapheneOS
with a locked bootloader."

## 1. Produce a real attestation chain

`KeystoreEd25519Signer.generateKeyPair` adds
`.setAttestationChallenge(challenge)` (32 random bytes minted at key-gen, stored
alongside the key so the verifier can confirm the leaf binds it). With a
challenge, `getCertificateChain(alias)` returns the full chain instead of a lone
self-signed leaf. The challenge is a one-time freshness/binding nonce: the key
is persistent, so it is attested ONCE at generation.

> **Correction (2026-06-04, on-device finding).** An earlier draft said "the
> chain stays valid for the key's life." That is wrong: the leaf is long-lived
> (epoch→2048), but the KeyMint **batch intermediate** that issues it is
> SHORT-LIVED — on the validation Pixels it was valid only `2026-06-01 →
2026-06-14` (≈2 weeks). Re-walking the stored chain weeks later therefore
> fails its validity-window check for a key that is genuinely hardware-backed.
> The resolution (landed) is the **verify-once-and-persist** model: verify the
> live chain near key-generation (the intermediate is fresh), then persist the
> resulting verdict (`DeviceAttestation`) in `cairn-storage`; later launches
> reuse the stored verdict rather than re-verifying an aged chain. The Rust
> verifier checks validity against the `now_unix` the caller passes, so it stays
> a pure function; the freshness policy lives in the Kotlin caller.

## 2. The Rust verifier (Stage 1)

A new `cairn_identity::attestation` (name TBD) exposes
`verify_key_attestation(chain: &[Vec<u8>], expected_challenge: &[u8], now_unix)
-> Result<AttestationFacts, AttestationError>` that:

1. Parses each DER cert; verifies leaf→…→root **signatures** with `x509-parser`'s
   `ring`-backed `verify_signature`, and validity windows against `now_unix`.
2. Checks the chain terminates at the **pinned Google Hardware Attestation
   Root** (compare the root's SPKI to the bundled pin).
3. Extracts the leaf's `1.3.6.1.4.1.11129.2.1.17` extension + ASN.1-decodes the
   `KeyDescription`: `attestationSecurityLevel`, `attestationChallenge`, and the
   `hardwareEnforced` `AuthorizationList` (`origin`, `purpose`).
4. Asserts: `attestationChallenge == expected_challenge`; `securityLevel ∈ {TEE,
StrongBox}`; `origin == GENERATED`; `purpose ⊇ {SIGN}`.

Returns `AttestationFacts { security_level, origin_generated, purpose_sign,
verified_boot: Option<…> }`. Exported via `cairn-uniffi` so the shell calls it
with the `attestation_chain` bytes; the result is surfaced on My-Identity
("Device key: hardware-attested ✓ — TEE").

**Test vectors:** capture a REAL chain from the Pixel (once §1 lands) as the
positive fixture; tamper cases (wrong root, bad signature, wrong challenge,
software security level) as negatives. The ASN.1 KeyMint decode is the
highest-risk surface — it gets the most test coverage.

## 3. GrapheneOS verified-boot (Stage 2 — D0020 §3.9)

The `AuthorizationList` carries a `RootOfTrust` (`verifiedBootKey`,
`deviceLocked`, `verifiedBootState`). Stage 2 pins the GrapheneOS
`verifiedBootKey` (its AVB signing key) and requires `verifiedBootState =
Verified` + `deviceLocked = true` — proving not just hardware-backed keys but a
locked GrapheneOS boot chain. Stage 1 parses + surfaces these advisorily; Stage
2 makes them an asserted pin. Split out because pinning the GrapheneOS AVB key +
handling the multi-device fleet is its own decision.

## 4. Failure posture — advisory, not a brick

Attestation is a **trust signal**, not an availability gate. The app already
falls back to a software key when no hardware key is available (it must never
brick on a provisioning gap, `generateOrLoad` returns null). So a failed/absent
attestation → the device key is recorded as "unattested" and the UI shows that
honestly; messaging still works. A future tier could REQUIRE attestation for
high-assurance contacts, but v1 surfaces the fact rather than gating on it.

## 5. Migration + revocation

- **Migration:** the in-place device key (no challenge) can't retroactively gain
  an attestation extension — adding the challenge regenerates it → a new op key
  → **fresh installs only**, the same migration the StrongBox-KEK + encrypted-DB
  changes carry (D0026 §12). Existing installs keep their unattested key until
  reinstall.
- **Revocation:** Google publishes an attestation-key revocation list (online).
  v1 pins the root only; a bundled-offline revocation snapshot (like the Sigsum
  release-log offline proof, D0024 §5) is a documented follow-up.

## Staging

- **Stage 1 — LANDED (2026-06-04).** §1 real challenge + §2 Rust verifier
  (`cairn_identity::attestation::verify_key_attestation`: chain → pinned Google
  root + KeyMint extension: security level, origin, purpose, challenge) + the
  uniffi export (`verify_device_key_attestation` → the advisory
  `AttestationResultRecord`) + the `CairnSession` wiring (verify-once-persist)
  - the My-Identity surface ("Device key: hardware-attested ✓ — TEE"). The
    verifier carries a positive vector (the real captured chain) + tamper
    negatives, including an rcgen-built **`RootNotPinned`** case (a fully
    self-consistent chain with an un-pinned self-signed root) and hand-built
    `KeyDescription` decode cases. On-device (two Pixel 6 / GrapheneOS): a real
    chain verifies to the pinned Google root with `securityLevel = TEE`; the
    advisory `RootOfTrust` surfaced `deviceLocked = true` with
    `verifiedBootState` = `SelfSigned` (device A) / `Verified` (device B) — i.e.
    the parser reads real-world variation correctly. Restart reuses the persisted
    verdict (no chain re-walk).
- **Stage 2 (follow-up):** §3 GrapheneOS verified-boot key pin + `deviceLocked` /
  `verifiedBootState` assertion (D0020 §3.9); §5 offline revocation. The Stage 1
  verifier already RETURNS the leaf Ed25519 device key + the `verified_boot_key`
  in `AttestationFacts`, so Stage 2's device-key binding + AVB-key pin can build
  on the existing surface without re-parsing.
