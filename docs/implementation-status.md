## Status legend

| Status           | Meaning                                                                                                                                                                                                                              |
| ---------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **IMPLEMENTED**  | The defense exists in code, has tests asserting the property, and verifies end-to-end under the gates (`cargo clippy --workspace --all-targets --all-features -- -D warnings`, `cargo fmt --all --check`, `cargo test --workspace`). |
| **PARTIAL**      | Some of the defense exists in code; gaps are named explicitly with the deferred work each gap blocks on.                                                                                                                             |
| **ASPIRATIONAL** | The design brief specifies the property but no code exists yet. Not a bug; just the honest state.                                                                                                                                    |

## Trust roots (design brief Â§3.4)

The brief explicitly lists trust roots as out-of-scope for the product to defend against. They are listed here only to confirm the boundary lines match the implementation:

- GrapheneOS, Pixel hardware, cryptographic primitives, Tor, SimpleX/Briar, Sigsum, build supply chain â€” all trusted by the implementation as the brief specifies. The Rust core does not attempt to defend against compromise of any of these.
- In-person facilitator (v1 pilot), reviewer pool (v1.5+), OIDC provider, witness-pool independence, reviewer toolkit, OOB incident-response channel, partner advisory authority â€” all operational trust placements; no Rust-core defense intended.

## What this status reveals about D0022 (storage layer)

Rows above that block on D0022 â€” and would unblock once a storage architecture decision lands:

1. **90-day stale-flag escalation** (`cairn-trust-graph` extension): per-attestation timer across sessions, requiring on-disk state.
2. **Trust-graph history persistence**: `verify_chain_links` currently consumes an in-memory slice; in production the chain is reconstructed from on-disk state.
3. **At-rest message persistence with passphrase + hardware-element binding**: the core stateful surface of the messaging layer.
4. **Recovery-peer share storage** per D0005: each peer holds one share + the matching commitment; both are on-disk.
5. **Atomic re-split coordination per D0018 Â§3.5**: two-phase commit across N peers requires both storage (intermediate state) and network (coordination protocol).
6. **Soft-flag UI state** (which attestations the user has acknowledged vs. left to auto-quarantine after the 90-day timer): per-attestation UI-decision history.

Items 1, 2, and 6 are pure storage-layer concerns. Items 3, 4, and 5 are storage + network combined; the storage decision is the prerequisite.
