# Implementation status vs. design brief

This document reconciles what is **actually implemented** against what the [design brief](design-brief.md) claims about Cairn's defenses. Its purpose is honesty: every defense the brief promises maps to one of five states below, with code references for the implemented ones.

This reconciles the v0.1.0 pilot pre-release tree. It is not the threat model — that lives in design-brief.md §3. It is also not a roadmap — that lives in design-brief.md §6 and §7. It is a snapshot of where the code is as of the commits referenced at the bottom of this file. Its bearing on the open decisions:

- D0011 self-audit and D0015 release-security posture are reconciled into the rows below; the residual release-security work splits into a v1 commitment (the witness pool + release-time Sigsum emit) and v1.5 items (the reproducible-builds pipeline + the recruited reviewer pool).
- D0022 (storage layer decision) is still open: the "DEFERRED" rows below that name it block on a storage architecture decision.

## Status legend

| Status           | Meaning                                                                                                                                                                                                                              |
| ---------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **IMPLEMENTED**  | The defense exists in code, has tests asserting the property, and verifies end-to-end under the gates (`cargo clippy --workspace --all-targets --all-features -- -D warnings`, `cargo fmt --all --check`, `cargo test --workspace`). |
| **PARTIAL**      | Some of the defense exists in code; gaps are named explicitly with the deferred work each gap blocks on.                                                                                                                             |
| **ASPIRATIONAL** | The design brief specifies the property but no code exists yet. Not a bug; just the honest state.                                                                                                                                    |
| **DEFERRED**     | Explicitly out of v1 implementation scope per the linked D-doc, usually because it depends on an unmade architectural decision (storage layer, network adapters, FFI surface) or on a v1.5+ feature lever.                           |
| **OUT-OF-SCOPE** | The design brief explicitly names this as never-in-scope (trust roots, indefinitely-out items per §3.5). Listed for completeness so readers can confirm the boundary lines match.                                                    |

A few Status cells carry **qualifiers**, not new states. Parenthetical or `+ proof` suffixes (e.g. `IMPLEMENTED + ON-DEVICE-PROVEN`, `ASPIRATIONAL (v1 commitment)`) just attach Evidence-level detail to one of the five states above. Bare `LANDED` reads as **IMPLEMENTED**; bare `OPERATIONAL` denotes a measure outside the Rust core (read it as **OUT-OF-SCOPE** for the core, with the operational owner named in the Evidence cell).

## Trust roots (design brief §3.4)

The brief explicitly lists trust roots as out-of-scope for the product to defend against. They are listed here only to confirm the boundary lines match the implementation:

- GrapheneOS, Pixel hardware, cryptographic primitives, Tor, SimpleX/Briar, Sigsum, build supply chain — all trusted by the implementation as the brief specifies. The Rust core does not attempt to defend against compromise of any of these.
- In-person facilitator (v1 pilot), reviewer pool (v1.5+), OIDC provider, witness-pool independence, reviewer toolkit, OOB incident-response channel, partner advisory authority — all operational trust placements; no Rust-core defense intended.

## What this status reveals about D0022 (storage layer)

Rows above that block on D0022 — and would unblock once a storage architecture decision lands:

1. **90-day stale-flag escalation** (`cairn-trust-graph` extension): per-attestation timer across sessions, requiring on-disk state.
2. **Trust-graph history persistence**: `verify_chain_links` currently consumes an in-memory slice; in production the chain is reconstructed from on-disk state.
3. **At-rest message persistence with passphrase + hardware-element binding**: the core stateful surface of the messaging layer.
4. **Recovery-peer share storage** per D0005: each peer holds one share + the matching commitment; both are on-disk.
5. **Atomic re-split coordination per D0018 §3.5**: two-phase commit across N peers requires both storage (intermediate state) and network (coordination protocol).
6. **Soft-flag UI state** (which attestations the user has acknowledged vs. left to auto-quarantine after the 90-day timer): per-attestation UI-decision history.

Items 1, 2, and 6 are pure storage-layer concerns. Items 3, 4, and 5 are storage + network combined; the storage decision is the prerequisite.
