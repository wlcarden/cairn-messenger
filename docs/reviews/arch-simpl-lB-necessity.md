# Architecture Simplification — Component-by-Component Necessity Lens

## Summary

Cairn v1 declares itself "the smallest cut of the architecture that delivers a defensible product" (§6.1:589) and absorbs 9-12 person-months of solo-developer effort to ship it. The component-by-component necessity review finds that this scoping claim is largely defensible for the cryptographic core — three-tier identity, trust-graph operation semantics, cascade quarantine, social recovery with layered peer verification — because each of these defends a Section 3 surface that the v1 audience and threat tier (§2.2, §3) cannot tolerate weakening. The integration argument in §4.3 (commitments compound rather than add) holds for those load-bearing components: removing any of them produces a degraded product whose security properties do not hold.

However, v1 includes at least three components that fail the "necessity for v1 audience and threat tier" test: the project-operated SimpleX crash-reporting queue, the multi-target build pipeline carrying cross-compilation scaffolding for v2/v3 platforms v1 does not ship, and the property-based migration test framework that tests round-trips between a single shipping schema and nothing. Two more components — UnifiedPush integration even off-by-default, and Sigstore-vs-long-lived-APK two-layer signing — could be defensibly deferred or simplified without weakening the v1 threat model, though both have honest forward-compatibility arguments. The reviewer-pool 3-of-5 attestation and Sigsum witness pool are each individually defensible but are gated on funding/recruitment paths (§8.2, §8.6, Q3, Q5) that the brief itself acknowledges may not close in time for v1 ship; the v1 ship-or-defer decision around them is the most honest cut available and is currently underacknowledged as such.

## Critical findings (components to cut from v1)

### F1: Project-operated SimpleX crash-reporting queue — cut entirely

- **Current commitment:** §5.7:580 and §6.1:607 commit to opt-in encrypted crash reporting "delivered through the existing messaging layer to a Cairn-team-controlled SimpleX queue." §6.1:607 acknowledges this is "the one acknowledged exception to the §4.2 minimal-project-operated-infrastructure principle, named explicitly here so the exception is visible." v1 includes "consent flow at provisioning, encrypted report delivery, and intake/triage infrastructure."
- **What v1 loses if cut:** A structured telemetry channel from pilot users to developer during the 9-12 month implementation period and the gated pilot deployment (per [D0013](decisions/D0013-pilot-consent-exit.md)). Crash data must instead reach the developer through the partner-mediated consent and exit protocol (D0013) — which is the channel through which substantive pilot feedback is already routed.
- **What v1 gains if cut:**
  - Removes the explicit §4.2 exception that the brief had to specifically carve out — the principle becomes operative without an exception (cleaner architectural story for funders and reviewers).
  - Eliminates one project-operated piece of infrastructure (queue creation, triage tooling, retention policy, key management for the project-side queue identity, eventual key rotation as the project moves from developer-personal to foundation-held per §8.4).
  - Removes consent-flow UX work at provisioning (§5.6's "informed-consent during onboarding" framing already carries multiple consent moments; one fewer reduces cognitive load on the pilot facilitator).
  - Removes the audit-scope item: the crash queue's intake/triage infrastructure is in scope for any audit that covers "release-security stack" (§8.5:809), expanding the pre-beta audit surface that §10.2 already finds expensive.
- **Recommendation:** Cut from v1. Pilot user feedback (including crash signals) flows through the §6.3/D0013 partner-mediated channel. Re-evaluate for v1.5 once pilot scale and pilot-feedback gaps make the case for a structured channel concrete. The §5.7:580 architecture is preserved as a v1.5 candidate; v1 does not pay implementation, consent-UX, audit-surface, and operational cost for a feature whose value at pilot scale (low double-digit users per §6.3) is unclear.

### F2: Multi-target build pipeline (cross-compilation scaffolding for v2/v3 platforms) — defer cross-compile scaffolding to v2

- **Current commitment:** §6.4:693 commits to "Build system designed to produce multiple artifacts even though v1 ships only one. The v1 build pipeline produces the GrapheneOS-Pixel APK. The same pipeline is designed to accept additional targets — the v2 USB image, the v2 iOS bundle, the v3 mesh-node firmware — without restructuring." The brief itself flags this as "a v1 architectural commitment with real engineering cost (keeping cross-compilation paths clean from day one, separating platform-specific code from the Rust core, designing the build matrix and CI configuration to accommodate platforms v1 does not ship for) rather than zero-cost 'engineering hygiene.'"
- **What v1 loses if cut:** v2 USB form factor and v2 iOS work (~12-18 months post-v1 per §6.2) will require build-system restructuring at that time rather than only target-addition. The §6.4 forward-compatibility promise around build is weakened from "additive" to "restructure-then-add."
- **What v1 gains if cut:**
  - The brief's own honesty note (§6.4:693-695) names this as a "dedicated v1 engineering line-item rather than zero-cost architectural commitment" absorbed into the 9-12 person-month estimate. Cutting it materially reduces v1 implementation scope.
  - The Rust-core architecture (§5.7) already provides the structural separation that v2 portability depends on — the build-pipeline cross-compilation work is in addition to that, not what enables it. A v2 USB form factor compiling the same Rust core to a different target uses Rust's cross-compilation natively; the v1 CI matrix and build-script structure that supports multiple targets simultaneously is independent v1 work.
  - Aligns with §10.1 Phase A's "as available" cadence: deferring optional scaffolding is exactly the kind of cut Phase A's volunteer-baseline framing implies.
- **Recommendation:** Cut multi-target build scaffolding from v1. v1 ships a single-target build pipeline producing the GrapheneOS-Pixel APK. The v1 Rust core (§5.7) preserves the forward-portability property the v2 work depends on. v2 work begins with build-system restructuring; the brief should acknowledge this is one additional engineering item v2 absorbs rather than a v1 commitment that makes v2 cheaper. The §6.4 forward-compatibility framing is honest only if the v1 cost is honestly accounted; cutting this item lets §6.4 say "v1 commits the things that genuinely cost nothing to commit, and defers the things that cost real v1 engineering."

### F3: Property-based migration test framework — defer to v1.5 when there is a v1→v1.5 migration to test

- **Current commitment:** §5.7:574 commits to "SQLite with explicit schema versioning and migration tests using property-based round-trip verification through each migration step." §6.4:691 reinforces: "Persistent storage (per Section 5.7) uses explicit schema versioning with migration tests that round-trip data through each migration step." §6.4:695 explicitly names "the property-based migration test framework per §5.7 (which is a dedicated subsystem, not free)" as a dedicated v1 engineering item absorbed into the 9-12 person-month estimate.
- **What v1 loses if cut:** Tests that exercise schema migration round-trips. But v1 has one schema; there is no migration to round-trip. The framework's first actual use is v1→v1.5.
- **What v1 gains if cut:** A "dedicated subsystem" (§6.4:695's own framing) of v1 implementation work that has no v1 test target. Property-based testing infrastructure for migrations is meaningful when there is more than one schema version; v1 ships with one schema. The schema-versioning fields (field-version, table-version markers) ship in v1 as designed; the property-based framework that exercises migrations is the v1.5 line-item.
- **Recommendation:** Cut the property-based migration framework from v1. v1 ships with schema-versioning fields present (the forward-compatibility property §6.4:691 requires) and plain schema definitions. v1.5's first concrete schema change (likely the trust-graph caching layer per §5.2:416's "v1.5 adds local caching") is when the property-based framework lands, against an actual migration to test. The §6.4:695 "dedicated subsystem" framing is honest about cost but does not establish v1 value.

## Significant findings (components to simplify or scope down)

### F4: Trust graph with five operation types — keep, but the "extension to additional types" architectural slot is overscoped for v1

- **Current commitment:** §5.2:367-373 specifies five operation types (attestation, attestation withdrawal, key compromise revocation, introduction, key rotation) with cascade quarantine semantics that explicitly close the cascade-laundering attack identified in the §5 adversarial review. §5.2:374: "Five operations are sufficient for the v1 graph; protocol versioning (Section 6.4) leaves room for additional types as later work identifies them, and unknown operation types are retained verbatim by older clients per the forward-compatibility property."
- **Necessity verdict:** The five types are individually necessary. The withdrawal vs. revocation split is what closes the cascade-laundering attack (§5.2:396), and removing it would weaken the trust-graph security model in a way the §3 threat tier cannot tolerate. The introduction operation is what propagates trust through the social network (§4.3:307); cutting it eliminates the graph-shape property that distinguishes Cairn from key-continuity systems. The key-rotation operation is what binds rotation events cryptographically to the master rather than asserting them through the previous operational key (§5.2:372).
- **What can be simplified:** The "unknown operation types retained verbatim" forward-compatibility property (§6.4:689) requires v1 clients to implement operation-handling code paths for operation types they do not understand. v1 can ship with the schema-versioning fields present (so v1.5 can add types) and simply reject unknown operation types in v1 rather than retain-and-forward them. v1.5 introduces retain-and-forward as part of its first actual operation-type addition. This is a small simplification but is the kind of "real engineering cost named as architectural commitment" that the brief's own §6.4:695 framing invites scrutiny of.
- **Recommendation:** Keep all five operation types and cascade semantics — these are load-bearing for the §3 threat tier. Simplify the retain-and-forward-unknown-types code path: ship v1 with "unknown types rejected" and add retain-and-forward in v1.5 alongside the first actual extension. Small but defensible cut.

### F5: Sigstore + long-lived APK key dual-signing — consider whether v1 needs both layers

- **Current commitment:** §5.5:502 commits to two signing layers: a long-lived APK signing key for "APK signature continuity (required for updates to install over prior versions)" and per-release Sigstore identity-based signing with Rekor logging. §5.5:520 documents both compromise responses, with the APK-key compromise specifically requiring multi-release APK Signature Scheme v3 rotation.
- **Necessity verdict:** The APK signing key is non-cuttable — Android's installer requires it for update continuity, and there is no operational alternative within Android's package model. The Sigstore-per-release signing is the cuttable layer.
- **What v1 loses if Sigstore signing is deferred to v1.5:** Per-release transparency-logged attestation through Rekor. The release-verification path becomes "APK key + reviewer Sigsum attestations" rather than "APK key + Sigstore-via-Rekor + reviewer Sigsum attestations."
- **What v1 gains if Sigstore signing is deferred:**
  - Removes the OIDC trust placement explicitly named in §5.5:510 (the U.S. jurisdiction concern, the hardware-security-key requirements, the alert-on-token-issuance operational discipline, the "Q24" open question).
  - Reduces audit scope for the pre-pilot audit (§8.5, [D0011](decisions/D0011-audit-budget-and-timing.md)): D0011 currently scopes the audit to cryptographic primitives, not release security; deferring Sigstore means release security stays out of audit scope through v1.
  - Reviewer attestations recorded in Sigsum already provide the per-release-event transparency log that §5.5:512 ("Public transparency log of releases") commits to. The release log entry is the reviewer attestation; the client refuses to install a release lacking quorum reviewer attestations in Sigsum. The Sigstore-via-Rekor layer is a second transparency anchor in addition to this.
- **Recommendation:** Consider deferring per-release Sigstore signing to v1.5 alongside the reviewer transition from source-review to binary-equivalence. The argument against deferral is the §4.3:309 differentiation framing — "Sigstore identity-based per-release signing on top of the long-lived APK key" is part of the architectural differentiation. The argument for deferral is that the reviewer-pool Sigsum attestations are already the operative transparency anchor for v1 release verification, and the second layer's value is partially the forward-compatibility property the brief itself would gain by waiting to commit until the OIDC jurisdiction question (Q24) resolves. This is a defensible v1.5 cut with named tradeoffs.

### F6: Sigsum witness pool — keep architecturally, but acknowledge the v1 ship-or-defer decision more directly

- **Current commitment:** §5.5:512 commits witness cosigning of release log state by "the same NGO and academic partners that cosign the trust-graph log state." §6.1:609 acknowledges witness pool recruitment is "conditional per Q5 outreach." §5.5:512 acknowledges the shared-witness-pool correlation risk and accepts it for v1.
- **Necessity verdict:** Sigsum witness cosigning is what closes the log-operator-tampering surface (§5.5:512). Without witnesses, the log operator (Sigsum project) is itself a single trust placement that can be coerced or compromised. The witness pool is what makes the log "publicly verifiable" rather than "trusted to one operator."
- **What v1 loses if cut:** The "publicly auditable via Sigsum commitments and witness cosignatures" property in §4.3:305 weakens to "publicly auditable via Sigsum commitments." The trust-graph public-audit property the §4.3 differentiation argument rests on degrades by one layer.
- **What v1 gains if cut:** The witness-pool recruitment dependency (Q5, §8.6) is removed from the v1 ship gate. §6.1:609 currently states "v1 release shipping is gated on first-quorum attestation forming; if the volunteer pool does not form in time, v1 ships are delayed at the median 4-6 month 'as quorum forms' cadence per [D0008]." The same gating applies to the witness pool.
- **Recommendation:** Keep the witness pool architecturally — Sigsum without witnesses degrades to a single-operator log, which §3.4 does not tolerate. But the brief should more directly acknowledge that the reviewer pool and the witness pool together constitute two recruitment gates on v1 ship, and that the absence of either is a v1-ship blocker per §6.1:609. The current framing treats both as "pool forms or ship delays" with no separation between them; a sharper acknowledgment would say "v1 ship requires both pools to form; if either is blocked, v1 ships in a degraded posture (without that layer) with explicit acknowledgment in release notes, or delays indefinitely." This is not a cut but a sharper acknowledgment of an existing dependency.

### F7: Reviewer pool 5-of-N → 3-of-5 attestation — keep, but the threshold's relationship to v1 ship discipline is underspecified

- **Current commitment:** §5.5:506 commits to 5+ recruited reviewers with 3-of-5 attestation threshold. §8.2 covers recruitment, honoraria, and "the operational policy for shipping below quorum (the project does not ship below 3-of-5 attestation; releases wait for quorum)."
- **Necessity verdict:** The multi-party reviewer-attestation property is what substitutes the developer's single signing identity with a distributed trust set (§4.3:309); cutting it returns release security to "trust the developer's word" plus reproducible builds. For v1, which ships before reproducible builds (§5.5:504 defers them to v1.5), the reviewer pool is the only multi-party check on the developer's source. Cutting it weakens v1 release security in a way the §3 threat tier does not tolerate.
- **What v1 could simplify:** The 5+/3-of-5 commitment could be initially relaxed to 3+/2-of-3 at v1 ship and tightened to 5+/3-of-5 at v1.5 alongside the reproducible-builds transition. This trades against the recruitment dependency (smaller pool is easier to recruit) and against the threshold property (2-of-3 has the same honest-majority property as 3-of-5 but smaller margin against single-reviewer attrition).
- **Recommendation:** Consider a v1-ship threshold of 3+/2-of-3 with a v1.5 tightening to 5+/3-of-5 alongside the reproducible-builds transition. The honest-majority property is preserved; the margin against attrition is reduced; the recruitment gate (§6.1:609) is softened. This is a defensible simplification but trades off the §4.3 differentiation framing — a smaller pool weakens the "no single jurisdiction can compel all" property §5.5:506 names. The brief should acknowledge the v1-ship threshold and recruitment-gate tradeoff explicitly rather than treating 5+/3-of-5 as the only target.

## Minor findings

### F8: UnifiedPush off-by-default polling — keep architecturally, the v1 cost is small

- **Current commitment:** §5.4:478-480 and §6.1:605 commit to UnifiedPush integration with v1 default off and polling-based delivery. The user opts in at provisioning if they want push.
- **Necessity verdict:** UnifiedPush integration code (distributor selection, NTFY-or-self-hosted endpoint configuration, the opt-in flow) is real implementation work even off-by-default. Polling at user-configurable intervals (default 15 minutes per §6.1:605) is independent of UnifiedPush and is the actual v1 default delivery mechanism.
- **What v1 could simplify:** Ship v1 with polling-only and no UnifiedPush integration; defer UnifiedPush to v1.5. v1 users either accept the 15-minute polling default or configure tighter polling. v1.5 adds UnifiedPush with the same opt-in-at-provisioning framing.
- **Recommendation:** Defensible v1.5 cut but small impact. The UnifiedPush integration is not load-bearing for v1 — polling-only is a coherent v1 posture and arguably more consistent with §4.2's minimal-infrastructure principle (UnifiedPush distributors are third-party metadata channels per §5.4:478, even when self-hosted). The v1 gain is modest implementation savings; the v1 loss is users who want push are told "polling-only until v1.5." Either choice is defensible; if v1 is genuinely scope-constrained, this is a cuttable line-item.

### F9: Capability tokens with arbitrary scope strings — keep, the cost vs. the closed-enumeration alternative is genuinely small

- **Current commitment:** §5.1:339-341 and §6.4:683 commit to COSE_Sign1 capability tokens with arbitrary scope strings rather than a closed enumeration.
- **Necessity verdict:** A closed enumeration (single boolean per device, or a fixed set of permissions) cannot express the v1 scope vocabulary (`messaging-send`, `trust-graph-attest`, `recovery-receive` — §5.1:339) without giving every device every permission, which defeats the scope-bounded property §5.1:343 commits to: "a device with a `messaging-send` token cannot issue trust graph attestations even if its key material is fully extracted." This is a v1 security property, not a forward-compatibility property.
- **What "arbitrary scope strings vs. closed enumeration" trades:** The cost difference is small — v1 ships with a documented set of scope strings either way; the difference is whether the COSE_Sign1 envelope's `scope` field is typed as `String[]` or as a closed enum. The forward-compatibility property of arbitrary strings (new scopes in v2/v3 without protocol change) is genuine but is the smaller of the two arguments for keeping it; the v1 security property (scope-bounded device authority) is the load-bearing argument.
- **Recommendation:** Keep. The "single boolean per device" alternative would lose the v1 scope-bounded property and weaken the §5.1:343 "compromised device cannot exceed token scope" guarantee. The arbitrary-string vs. closed-enum choice is a small cost; the device-authority-as-boolean alternative is a real weakening.

### F10: Three-tier identity — keep, the rotation-as-tier-separation alternative weakens the master-cold property

- **Current commitment:** §5.1:323-361 specifies three tiers: master (Shamir-split, off-device), operational (hardware-keystore-held, rotatable), device (short-lived capability tokens).
- **Necessity verdict:** §5.1:345-351 already evaluates the alternatives. A single rotatable identity with rotation as tier-separation would mean the active signing key is also the user's stable identity; rotation under coercion (§5.1:333) would reset the stable identity, breaking the trust-graph anchor that §5.2 depends on. The §5.1:347 analysis ("device compromise — which is well within the threat model — compromises the identity that the trust graph anchors on, with no clean path to recover without losing all attestations") applies directly to the single-rotatable-identity alternative.
- **What v1 could simplify:** Nothing without weakening the trust-graph anchoring property. The three-tier model is the simplest construction that delivers (master cold + operational rotatable + device scope-bounded).
- **Recommendation:** Keep. The alternative would weaken §5.2's trust-graph-anchored-in-stable-identity property.

### F11: Pre-shared peer challenges + 48-hour delay-and-confirm — keep both, the layered-resistance framing depends on both

- **Current commitment:** [D0005](decisions/D0005-peer-verification.md) and §5.3:426-430 commit to both pre-shared peer challenges and a 48-hour delay-and-confirm window.
- **Necessity verdict:** §5.3:440 frames this explicitly: "an adversary who has identified the peers must additionally obtain the per-peer challenges and either prevent the user from cancelling during the cooling-off window or persuade peers to release shares without challenge verification." Each mechanism defends a different surface — challenges defend against impersonation (the user-was-not-the-one-asking case); the cooling-off window defends against continuous-control coercion (the user-asked-while-coerced case). Cutting either reduces the layered resistance to a single layer.
- **What cutting the challenge would cost:** Voice cloning, social engineering of the peer, and impersonation of the recovery request through the messaging channel all become single-step attacks rather than two-step. §5.3:426 explicitly: "the phrases never travel through the recovery-request channel; they exist for the peer to challenge the user with at share-release time."
- **What cutting the delay-and-confirm would cost:** A continuous-control adversary (who has the user, the device, and full unlock for the brief period needed to produce challenge phrases under coercion) reaches the master in a single coordinated event with no window for the legitimate user to cancel. §5.3:430 explicitly: "the legitimate user can cancel through any channel they control."
- **Recommendation:** Keep both. Cutting either weakens the layered-resistance property §5.3:440 names.

### F12: Multi-distribution channels (F-Droid + Accrescent + direct + Tor) — keep at least F-Droid + direct, Accrescent and Tor are defensible v1.5 cuts

- **Current commitment:** §5.5:516 commits to F-Droid, Accrescent (as the platform matures), GitHub releases, Tor onion service, and offline signed images. §6.1:609 names "F-Droid, Accrescent, and direct download from a project-controlled domain" as v1 channels.
- **Necessity verdict:** Multi-channel distribution is the defense against "channel-specific blocking by adversaries with the ability to interdict particular routes" (§5.5:516). Each channel is independent verification: a user receives the same Sigstore-signed Sigsum-logged artifact through any channel.
- **What v1 could simplify:** Ship v1 with F-Droid + direct download from project-controlled domain only. Accrescent is "as the platform matures" — defer to v1.5 when the Accrescent platform's pilot-relevant maturity is established. Tor onion service is real operational work (key management, onion endpoint operation, mirroring discipline); defer to v1.5. Offline signed images for hand-delivery are a v1 operational practice (the facilitator brings the APK on USB to the in-person provisioning ceremony per §6.1:601), not a distribution channel infrastructure commitment.
- **Recommendation:** Defensible simplification — ship v1 with F-Droid + direct as the two committed channels; Tor onion and Accrescent are v1.5. The §5.5:516 "users in jurisdictions where F-Droid is blocked obtain releases through Tor or offline images" framing is partially served by the offline-image facilitator workflow already committed in v1.

## Patterns

**Pattern 1: §6.4 forward-compatibility commitments are not zero-cost, and the brief acknowledges this but does not consistently apply the acknowledgment.** §6.4:693-695 honestly names the multi-target build pipeline and property-based migration framework as "dedicated v1 engineering line-items rather than zero-cost architectural commitments." This honesty creates a sharper question for each §6.4 item: does the v1 cost defend a v1 property, or does it merely preserve a v2/v1.5 option? F2 (multi-target build) and F3 (migration framework) are the two items where the cost is paid in v1 for a property that has no v1 use. F4 (retain-unknown-types-verbatim) is a smaller version of the same pattern. The brief's own framing invites the cut.

**Pattern 2: The "minimal project-operated infrastructure" principle has one named exception (§6.1:607 crash queue) and at least one unnamed near-exception (the witness pool and reviewer pool require project-led recruitment per §8.2/§8.6).** The named exception (F1, crash queue) is cuttable. The unnamed near-exceptions are not (they are §4.3:309 differentiation properties), but the brief should treat them with the same explicit-exception framing it applies to the crash queue: each is a project-operational dependency, and the v1-ship-or-degrade decision around each should be acknowledged at the §4.2 principle's level rather than buried in §6.1:609.

**Pattern 3: Several v1 components are load-bearing for the §3 threat tier and are correctly retained.** Three-tier identity (F10), trust-graph operation semantics with five types and cascade quarantine (F4 keep), the layered peer-verification mechanism (F11), scope-bounded capability tokens (F9), reviewer-pool attestation (F7 keep, possibly reduce threshold), and Sigsum witness cosigning (F6 keep) are the components the §4.3 "commitments compound" argument is correctly applied to. The simplification question for these is "tighten the threshold" or "soften the recruitment gate," not "cut entirely." The brief's scoping discipline is correct for these items.

**Pattern 4: The v1 ship gate is currently gated on three recruitment-dependent items (reviewer pool per §8.2, witness pool per §8.6 Q5, partner outreach per [D0013](decisions/D0013-pilot-consent-exit.md)) and the brief acknowledges this only partially.** §6.1:609 names the reviewer-pool gate; the witness-pool gate and partner-outreach gate are mentioned but not consolidated. A v1-ship checklist that names all three would be a small but meaningful tightening of §6.1, and would let the brief's forward-compatibility honesty (§6.4:693-695) extend to the v1-ship-conditions honesty the brief implicitly already commits to.
