# Architecture Simplification — YAGNI / Over-Engineering Lens

## Summary

The brief justifies its architecture honestly, but the volume of v1 commitments — five trust-graph operation types, nine-field signed envelope with prior-hash chain + issuer-cert-hash binding, cascade quarantine _plus_ 90-day stale-flag escalation, pre-shared challenges _plus_ 48-hour delay, multi-target build pipeline for platforms v1 does not ship, capability-token scope vocabulary for capabilities v1 does not have, crash-reporting infrastructure for ~12 users the developer can talk to in person, UnifiedPush distributor selection UX for a feature that ships off-by-default — has accumulated more architecture than a 10–15-user pilot needs and more than one volunteer-cadence developer can hold context for through ~9–12 person-months of implementation per §6.1. Several commitments are double-defenses against threats §3 acknowledges as residual surfaces anyway; several are forward-compatibility scaffolding for v2/v3 features the project may never reach (§10.8 explicitly disclaims Phase C/D); several exist to make the trust-graph design defensible to reviewers/funders rather than because a named v1 threat requires them.

The cuts below are calibrated to v1's actual audience (pilot users known to the developer, onboarded in person, with the developer as facilitator). Where I propose removing a commitment, I name the specific v1 adversary scenario that does _not_ become viable as a result, and where I propose deferring forward-compatibility scaffolding, I name the migration cost the deferral creates and argue that cost is smaller than the cost of building the scaffolding before the destination is known.

## Critical findings (cuts you'd make at v1)

### F1: Cascade quarantine + 90-day stale-flag escalation is over-machined for 10–15 users

- **What the brief commits to:** §5.2 specifies cascade-quarantine semantics on revocation (lines 396–400: attestation withdrawal soft-flags downstream; key-compromise revocation hard-suspends post-`revoked_as_of`, soft-flags prior) _plus_ a 90-day automatic stale-flag escalation that re-quarantines untouched soft-flags (§5.2:400). §6.1:597 commits both to v1.
- **Cost to v1 (engineering, audit scope, operational maintenance):** Two distinct path-computation modes (soft vs. hard), per-attestation timer state surviving app restarts/reinstalls, a quarantine-decision UX with three branches (re-attest / accept-with-acknowledgment / defer), and audit coverage on the 90-day clock's correctness (does inaction default to quarantine if the device was offline for 91 days? if Sigsum was unreachable? if the user re-installed?). The pre-pilot audit per [D0011] now has to cover the cascade state machine, not just the cryptographic envelope.
- **Use case justifying the commitment at v1:** §5.2:402 names the value: "user signals _severity_ — withdrawing endorsement of a contact is not the same operation as declaring their own key compromised." That signal matters at population scale where you cannot personally walk each user through a revocation. At pilot scale (10–15 users, developer-as-facilitator per §6.3) the developer can have the conversation directly when a peer revokes someone. The cascade-laundering attack §5.2:374 cites is real, but the laundering threat presupposes a population large enough that the user-re-attestation channel is the only feasible response — not a pilot where revocation events will be handled in-band between the developer and the affected users.
- **Cuts you'd make:** Ship v1 with **one** revocation operation type (the hard "key compromise" form) plus attestations, introductions, and rotations. Drop attestation withdrawal entirely from v1; if a user wants to retract an endorsement at pilot scale, they tell the developer-facilitator out-of-band. Drop the 90-day stale-flag escalation — without the soft-flag split there is nothing to escalate, and the warning-fatigue argument §5.2:400 cites is irrelevant at 15 users. Bring withdrawal + cascade + escalation back in v1.5 when broader-than-pilot release is the goal.
- **What would break (named v1 threat or audience case):** Not the cascade-laundering attack at pilot scale — the developer-facilitator's direct contact with each user closes that loop operationally rather than mechanically. What is lost is the ability for a user to express _"I don't trust X anymore but my key wasn't compromised"_ without a conversation; at pilot scale that conversation is cheap.
- **Recommendation:** Defer attestation-withdrawal-as-operation-type and 90-day stale-flag escalation to v1.5. Document the v1 simplification in [D0006] explicitly so reviewers see the scope cut. Net savings: meaningful cut to the v1 trust-graph state machine, removal of the timer-correctness audit branch, and ~1 person-month of v1 engineering relative to the brief's current §6.1 scope.

### F2: Pre-shared peer challenges OR 48-hour delay — not both — for v1's 10–15 users

- **What the brief commits to:** §5.3:426–430 commits to _both_ pre-shared per-peer challenges (established at provisioning, stored off-device by the user, stored encrypted on each peer's device) _and_ a 48-hour delay-and-confirm window before any share release. Per [D0005].
- **Cost to v1 (engineering, audit scope, operational maintenance):** Two independent verification mechanisms means two UX flows for peers (peer storage of challenge + peer challenge-prompt UI at recovery time; plus the 48-hour countdown UI on the fresh device showing cancel channels), two user-education modules at provisioning (the user must understand both the phrase exchange _and_ the cooling-off window), and two failure modes the recovery handbook must document. The pre-pilot audit per [D0011]:611 must cover the challenge storage format on peer devices as well as the cooling-off-window state machine on the recovering device.
- **Use case justifying the commitment at v1:** §5.3:440 calls the combination "layered resistance." Each layer attacks a different sub-threat: the pre-shared challenge defeats voice-cloning impersonation; the 48-hour delay defeats _continuous-control_ coercion only if the user is _not_ under continuous control for the full 48 hours. §3.2:148 explicitly assumes adversaries "with time horizons measured in months rather than minutes" — meaning the 48-hour window does not defeat the named threat tier on its own merits; the brief is candid about this at §4.3:307 ("the chain remains achievable for a continuous-control adversary who can hold the user 48 hours and extract per-peer challenges from on-device storage"). And the pre-shared challenge defeats voice-cloning impersonation without needing the 48-hour delay.
- **Cuts you'd make:** Ship v1 with pre-shared peer challenges _only_. The challenge alone closes the unlock-then-impersonate chain at the cost the threat tier cares about (the adversary needs material _not on the device_, which is the architectural property §5.3 is fundamentally selling). Defer the 48-hour delay-and-confirm to v1.5 when broader-than-pilot release introduces users the developer cannot personally contact during a recovery event.
- **What would break (named v1 threat or audience case):** A continuous-control adversary who _already has_ the per-peer challenges (extracted from password manager or from peer-device storage) would not be additionally delayed by the cooling-off window. The 48-hour delay matters most when the user is _not_ under continuous control and can cancel — which is, by §3.2's adversary capability framing, a narrow regime. At pilot scale, recovery events will be visible to the developer-facilitator in real time anyway; the developer can mediate cancellation through any peer.
- **Recommendation:** Ship pre-shared challenges in v1 (the load-bearing layer). Defer the 48-hour window to v1.5. Net savings: removal of the cancel-channel UI, the per-peer countdown display, the cancellation propagation logic, and the post-cancellation cleanup semantics. Re-introduce in v1.5 when the operational discipline of the pilot's developer-as-facilitator is no longer the backstop.

### F3: Issuer-cert-hash binding is post-hoc defensiveness, not load-bearing at v1

- **What the brief commits to:** §5.2:379 specifies an `issuer-cert-hash` field on every trust-graph operation — "a SHA-256 hash of the master attestation certifying the issuer's operational key. The signature commits to this hash, so the operation is cryptographically bound to a specific master attestation of the issuer key." [D0006] is cited as the source decision.
- **Cost to v1 (engineering, audit scope, operational maintenance):** A ninth signed field on every operation, an additional verification step in path computation (verifier must fetch and hash the issuer's master attestation before validating), schema-versioning cost on the COSE_Sign1 envelope, and audit-scope expansion on the per-(issuer, subject) anti-equivocation property.
- **Use case justifying the commitment at v1:** The brief's justification: "a verifier follows the chain operation → issuer's master attestation → user's master public key, with no implicit lookup ambiguity if the issuer has multiple master-signed operational keys (rotation overlap, etc.)." The named v1 scenario this defeats is rotation-overlap ambiguity — but §6.1:597 also says "proactive rotation flow may defer to v1.5" and §5.1's rotation moments (provisioning, post-coercion, suspected key misuse) are infrequent and ceremonial. At v1 pilot scale, rotation overlap is a rare edge case the developer-facilitator can mediate by hand if it arises.
- **Cuts you'd make:** Drop `issuer-cert-hash` from the v1 envelope; the verifier resolves the issuer key against the most recent master-signed operational key at the time of the operation's timestamp. Re-introduce the field in v1.5 when rotation cadence picks up and the population scale precludes hand-mediation.
- **What would break (named v1 threat or audience case):** A rotation-overlap ambiguity attack — where the issuer has two simultaneously valid master-signed operational keys and a verifier picks the wrong one — but this presupposes proactive rotation (deferred to v1.5 per §6.1:597) and a population where the verifier cannot ask the issuer to clarify. Neither holds at v1.
- **Recommendation:** Drop `issuer-cert-hash` from v1; ship the eight-field envelope. Document the field as a v1.5 addition aligned with the rotation-flow completeness commitment. Net savings: smaller v1 envelope, narrower pre-pilot audit scope, one fewer schema field to maintain forward-compatibility on.

### F4: §6.4 multi-target build pipeline is premature optimization for unshipped platforms

- **What the brief commits to:** §6.4:693 commits the v1 build pipeline to accept "additional targets — the v2 USB image, the v2 iOS bundle, the v3 mesh-node firmware — without restructuring," with the candid acknowledgment that this is "a v1 architectural commitment with real engineering cost (keeping cross-compilation paths clean from day one, separating platform-specific code from the Rust core, designing the build matrix and CI configuration to accommodate platforms v1 does not ship for)."
- **Cost to v1 (engineering, audit scope, operational maintenance):** The brief itself acknowledges this is non-zero engineering. CI matrix complexity, build-system abstractions that exist solely to accommodate iOS Swift integration v2 may or may not need, build-time platform-detection logic for mesh firmware that v3 may or may not need given the §7.1:715 acknowledgment that v3 depends on mesh-community partnerships not yet established.
- **Use case justifying the commitment at v1:** §6.4:693's argument is that retrofitting cross-compilation later is more expensive than designing for it from day one. This is the standard premature-optimization defense and it presupposes the project reaches v2/v3. §10.8 explicitly does not promise the project reaches Phase C or Phase D. §7.1:713–715 makes v2 and v3 conditional on Phase D funding and on partner relationships not yet established. The project may never need these targets.
- **Cuts you'd make:** Ship v1 with a single-target Android build pipeline. Keep the Rust/Kotlin boundary clean (which [D0003] already mandates for security reasons) but do not pre-architect the build system for platforms the project may never ship. The Rust core's cross-compilability is a _language_ property, not a _pipeline_ property; v1.5 can add iOS targets to the pipeline if v2 funding actually closes.
- **What would break (named v1 threat or audience case):** Nothing v1 needs. The cost of retrofitting the build pipeline in v2 is real but is borne against v2 funding (Phase D per §7.1:713), at which point the project has the resources to do it. Pre-building it in v1 is taking a Phase D cost out of the volunteer-baseline budget.
- **Recommendation:** Drop the multi-target build-pipeline commitment from §6.4. Keep the Rust/Kotlin separation (which serves v1 security per [D0003]); abandon the v2/v3 cross-compilation scaffolding. Net savings: weeks of v1 build-system work, narrower CI surface, and an honest framing that v2 platform expansion is v2 work, not v1 work.

### F5: Capability-token scope vocabulary is over-specified for v1's actual capabilities

- **What the brief commits to:** §5.1:341 and §6.4:683 commit to capability tokens with "arbitrary scope strings rather than a closed enumeration, so later versions can introduce new scopes (`mesh-relay` in v3, `usb-attached-device` in v2, and so on) without breaking compatibility with older clients." v1 ships with the scope strings `messaging-send`, `trust-graph-attest`, `recovery-receive`.
- **Cost to v1 (engineering, audit scope, operational maintenance):** Scope-string parsing logic, scope-validation rules, the conceptual framework of "unknown scope → decline," forward-compatibility audit branches.
- **Use case justifying the commitment at v1:** The justification is that v2 and v3 will introduce new scopes. At v1 single-device-per-identity per [D0007], there is _one device per identity_ and that device gets all three scopes. The scope mechanism is doing no work at v1 — the same device that sends messages also signs trust-graph operations and receives recovery shares. The architectural distinction exists for v2 USB form factor (where the USB might hold `recovery-receive` while the phone holds `messaging-send`) and v3 mesh-node (where the mesh node holds `mesh-relay`).
- **Cuts you'd make:** Ship v1 with a single scope-bundle (all-or-nothing capability tokens). The token still grants the device authority to act as the operational identity's delegate; it simply does not differentiate between message-sending and trust-graph-attesting _at v1 where the same device does both anyway._ Re-introduce scope strings in v2 when the USB form factor actually requires the distinction.
- **What would break (named v1 threat or audience case):** Per §5.1:343, "a device with a `messaging-send` token cannot issue trust graph attestations even if its key material is fully extracted by forensic tooling." But at v1 single-device, the device that has been forensically extracted _also has the `trust-graph-attest` token_. The scope distinction protects nothing v1 has. The post-extraction defense is operational-identity revocation through the trust graph, not scope distinction.
- **Recommendation:** Ship v1 with all-or-nothing capability tokens. Document the scope mechanism as v2 work that introduces alongside the multi-device extension already deferred per [D0007]. Net savings: one fewer schema dimension to specify and audit, simpler v1 token verification, and an honest acknowledgment that scope vocabulary is a multi-device feature.

## Significant findings (simplifications you'd recommend)

### F6: Crash reporting infrastructure for a pilot the developer can email

- **What the brief commits to:** §5.7 and §6.1:607 commit to opt-in, end-to-end encrypted crash reports delivered to a project-operated SimpleX queue, with intake/triage infrastructure. The brief acknowledges this as "the one acknowledged exception to the §4.2 minimal-project-operated-infrastructure principle."
- **Cost to v1 (engineering, audit scope, operational maintenance):** Consent-flow UI, encrypted-report serialization, project-side SimpleX queue operation (ongoing), intake/triage workflow, audit coverage of the consent boundary, and the principled exception itself becomes one more thing the brief has to defend to reviewers.
- **Use case justifying the commitment at v1:** "Crash reports help us debug." At pilot scale (10–15 users known to the developer per §6.3, onboarded in person, with developer-as-facilitator presence in §5.6), the developer's preferred crash-reporting protocol is "the user texts me." A dedicated infrastructure exists to scale crash reporting to a population the developer cannot personally know.
- **Cuts you'd make:** Drop the crash-reporting infrastructure from v1. Pilot users with crashes report them through the existing facilitator-mediated channel (in person, on the messaging layer the user is already running). Re-introduce crash reporting in v1.5 when broader-than-pilot release introduces users the developer does not know.
- **What would break:** Nothing v1 needs. The user-education cost ("how do I get my crash to the developer?") is trivial at pilot scale and is already implicit in the developer-as-facilitator relationship.
- **Recommendation:** Defer crash-reporting infrastructure to v1.5. Net savings: project-operated infrastructure cleaned up, one fewer principle-exception to justify, intake/triage work removed from the volunteer-baseline working set.

### F7: UnifiedPush distributor-selection UX for an off-by-default feature

- **What the brief commits to:** §5.4:480 and §6.1:605 ship v1 with push notifications off-by-default and polling at 15-minute intervals. The brief also commits to "UnifiedPush distributor selection UX" and an "opt-in flow during provisioning when the user elects active push delivery."
- **Cost to v1 (engineering, audit scope, operational maintenance):** UnifiedPush integration logic, distributor-selection UI, the facilitator-walks-through-implications scripting for provisioning, and audit coverage of the push-metadata boundary for users who opt in.
- **Use case justifying the commitment at v1:** The brief's justification is "users who explicitly opt into push at provisioning are walked through the implications by the facilitator." At pilot scale, push is off, polling works, the latency cost is acceptable, and pilot feedback is the explicit gate (§5.4:480) for revisiting the default.
- **Cuts you'd make:** Ship v1 polling-only. Drop the UnifiedPush opt-in path entirely until pilot feedback indicates polling latency is operationally unworkable. The architectural slot for push is preserved by the protocol abstraction; the _UX surface_ for selecting distributors is what gets cut.
- **What would break:** Nothing v1 needs. The brief itself frames v1.5 as the moment to revisit push (§5.4:480); building the opt-in path in v1 is paying for a feature whose default the brief has not committed to.
- **Recommendation:** Defer the UnifiedPush opt-in path to v1.5 when pilot feedback exists. Net savings: provisioning ceremony shortened (one fewer informed-consent module the facilitator walks through), distributor-selection UI deferred, push-opt-in audit scope deferred.

### F8: Sigstore long-lived APK key + per-release Sigstore identity is two-layer where v1 ships closed pilot

- **What the brief commits to:** §5.5:502 specifies two signing layers: a long-lived APK signing key (required by Android for signature continuity) _plus_ per-release Sigstore identity-based signing through Fulcio/Rekor. The brief commits to operational defenses on the OIDC identity (hardware key, alerts on issuance, regular Rekor audit) and acknowledges that "v1 uses a U.S.-based OIDC provider in pilot ... places U.S. legal process in the effective trust surface" (§5.5:510).
- **Cost to v1 (engineering, audit scope, operational maintenance):** Two signing flows per release, OIDC identity operational hardening, Rekor monitoring infrastructure, the explicit acknowledgment that U.S. legal process is in v1's trust surface — a posture §3.4 names but that an honest design might prefer to avoid by simply not adding the OIDC layer at v1.
- **Use case justifying the commitment at v1:** The brief's argument is that Sigstore's transparency log makes the developer's per-release signing event publicly auditable. At v1 pilot scale (10–15 users known to the developer, releases delivered through the developer-as-facilitator channel) the auditability is doing operational work that is already accomplished by the developer being personally present at install. The reviewer-attestation layer (3-of-5 of recruited reviewers) is the load-bearing supply-chain defense; Sigstore is an additional layer that adds U.S. legal-process trust surface for marginal audit value at pilot.
- **Cuts you'd make:** Ship v1 with the long-lived APK key (Android requires it) plus reviewer attestations in Sigsum (load-bearing). Drop the per-release Sigstore identity-signing layer from v1; add it at v1.5 when broader-than-pilot release means the developer is no longer personally present at install.
- **What would break:** The audit timeline-reconstructability of release-signing events would be marginally weaker at v1 — Rekor would not have the per-release entries. But the Sigsum log of reviewer attestations _is_ the cross-checkable timeline. Removing one of two cross-check layers does not collapse the defense; it removes redundancy.
- **Recommendation:** Defer Sigstore identity-based signing to v1.5. Document v1's release-security stack as long-lived APK key + reviewer attestations in Sigsum + transparency-log-via-Sigsum-only. Net savings: OIDC operational hardening deferred, U.S. legal-process trust surface removed from v1, Rekor monitoring infrastructure deferred, ~one trust root removed from §3.4 v1 enumeration.

### F9: Property-based migration test framework for unshipped schemas

- **What the brief commits to:** §5.7:574 and §6.4:691 commit to "SQLite with explicit schema versioning and migration tests using property-based round-trip verification through each migration step" — a dedicated subsystem the brief acknowledges is "a dedicated subsystem, not free" (§6.4:695).
- **Cost to v1 (engineering, audit scope, operational maintenance):** Property-based testing infrastructure, migration-test harness, the discipline of writing round-trip properties for every schema change, audit coverage of migration correctness.
- **Use case justifying the commitment at v1:** "v1.5 and later versions can extend the schema." v1 has _one_ schema version. The property-based migration framework is testing a migration path that does not exist yet, against a future schema that has not been specified yet.
- **Cuts you'd make:** Ship v1 with one schema version, no migration framework. When v1.5 needs a migration (and only then), write the migration plus its test. Property-based testing for migrations can be introduced at the moment there _is_ a migration to test.
- **What would break:** Nothing v1 needs. The brief's argument is that retrofitting migration discipline later is harder than designing for it from day one — but in practice, the first migration is when the discipline gets honed against real schema-change cases, not against speculative ones.
- **Recommendation:** Defer the property-based migration framework to v1.5 when the first real migration exists. Keep schema-versioning _fields_ in v1 (they cost ~nothing) but defer the _test framework_ until there is a migration to test. Net savings: one fewer v1 subsystem to build and audit; honest acknowledgment that migration discipline is incrementally accumulated.

## Minor findings

### F10: Loaner-pool inventory for the pilot

§10.1 / §6.3 commits to 2–4 loaner devices for users who cannot acquire hardware. At 10–15 users known to the developer, the developer can negotiate hardware case-by-case rather than committing to standing inventory. Drop the loaner pool as a commitment; treat hardware support as an ad-hoc developer cost at pilot. Net savings: ~$2k-$3k in pre-pilot capital outlay deferred until broader-release exists.

### F11: Voice and video calling slot preserved in v1 architectural language

§5.4:482 says voice/video "preserves the slot" but is v1.x/v2 candidate. The brief is already cutting it; the §5.4 paragraph is just leftover preservation language. Drop the preservation language; if v1.x ships voice/video, it ships against whatever architectural posture is current then. Net savings: one fewer slot-preservation commitment §6.4 has to honor.

### F12: §3.4 trust-root explicit enumeration of OIDC provider, Rekor, reviewer onboarding toolkit, out-of-band incident-response channel, partner advisory authority, future foundation jurisdiction

§3.4 enumerates _fourteen_ trust roots, several of which (OIDC provider, Rekor, reviewer onboarding toolkit, future foundation jurisdiction) exist solely because §5.5 architectural commitments require them. Cutting F8 above removes the OIDC + Rekor trust roots; cutting F1/F3 narrows reviewer onboarding toolkit scope. The §3.4 enumeration shrinks proportionally. Honesty about trust roots is a virtue, but adding architectural surfaces just so they can be enumerated in §3.4 is the wrong direction.

## Patterns

**Pattern 1: Defensive belt-and-suspenders against threats §3 acknowledges as residual.** F1 (cascade quarantine _plus_ stale-flag escalation), F2 (pre-shared challenges _plus_ 48-hour delay), and F8 (long-lived APK key _plus_ Sigstore identity) all combine two mechanisms against threats §3 already says are residual surfaces the architecture does not fully close. The combination produces audit complexity without closing the threat; the single load-bearing layer in each case is sufficient at v1 scale.

**Pattern 2: Forward-compatibility scaffolding for v2/v3 features whose funding §10.8 does not promise.** F4 (multi-target build pipeline), F5 (capability-token scope vocabulary), F9 (property-based migration framework), and parts of [D0007]'s remaining multi-device-schema commitment all exist to make v2/v3 work cheaper _if v2/v3 ships_. §7.1 and §10.8 make clear that v2 and v3 are conditional on funding and partner relationships the project does not control. Building the scaffolding before knowing the destination is the standard YAGNI mistake; the cost is paid against volunteer-baseline budget that §9.1 identifies as the binding constraint on v1.

**Pattern 3: Population-scale machinery deployed at pilot scale.** F1 (cascade + stale-flag), F6 (crash reporting infrastructure), F7 (UnifiedPush distributor-selection UX), and parts of F8 (Sigstore identity-signing's audit-trail value) all exist to handle populations the developer cannot personally know. At pilot scale (10–15 users known to the developer per §6.3) the developer-as-facilitator relationship per §5.6 is the operational backstop these mechanisms substitute for at scale. Building the substitutes _and_ keeping the facilitator presence is paying twice for the same property.

**Pattern 4: Trust-graph engineering optimized for reviewer/funder defensibility rather than v1 user value.** The nine-field envelope, the five-operation type system, the cascade-with-escalation, and the issuer-cert-hash binding all make the trust graph more defensible in an adversarial review than in the original §5 draft (per the §5 adversarial review F4/F11 findings the brief absorbs into D0006). The audit scope these additions create is paid against the same finite pre-pilot audit budget per [D0011]. The brief is buying defensibility with audit hours that could otherwise cover the load-bearing primitives (Shamir reconstruction, COSE_Sign1 envelope, recovery-flow memory hygiene). The trade is structurally backwards: prioritize audit coverage of the components a v1 user depends on, not the components a v1.5 user _would_ depend on.

---

## 150-word summary

The brief's v1 architecture has accumulated defensible commitments for population-scale, multi-platform, multi-device futures the project may never reach. At 10–15 pilot users known to the developer, with the developer present at install, several layered defenses double-cover the same threats §3 acknowledges as residual anyway (cascade + 90-day escalation; pre-shared challenges + 48-hour delay; long-lived APK key + Sigstore identity), and several forward-compatibility commitments (multi-target build pipeline, capability scope vocabulary, property-based migration framework, issuer-cert-hash binding) pay v1 engineering cost against unshipped v2/v3 features whose funding §10.8 explicitly does not promise. Cutting nine specific commitments (one per finding F1–F9) reduces v1's pre-pilot audit scope, narrows the §3.4 trust-root surface, removes U.S. legal-process from v1's trust surface, and reclaims volunteer-baseline engineering capacity for the load-bearing primitives. Each cut is reversible at v1.5 when broader-than-pilot release actually arrives.
